The Complexity Problem

Why we need architectural models in networking

Challenge: diverse hardware, varied software and multiple protocols must interoperate globally, built by different vendors over decades.

Without layering

  • Each application would have to understand each network technology
  • Changing the hardware would mean rewriting the software
  • No interoperability between vendors

result: Haos. Insule proprietare.

With layering

  • Each layer has focused responsibilities
  • Layers interact through well-defined interfaces
  • Components can be replaced without affecting other layers

result: Internet global. Trilioane of devices.

Analogy Constructiei

Understanding arhitecturii layerede through constructii

Separation of responsibilities: an electrician does not alter the foundations. An interior designer does not rewire the building.

  • Each specialist works at their own layer
  • Upper layers rely on services provided by lower layers
  • Changes in one layer do not require rebuilding the entire stack

In networking: a web developer can write HTTP without needing to understand optical fibre.

Layers 1 & 2 ‚Äì Physical and Data Link

Fundamentul comunicatiei in network

the level 1 ‚Äì Physical

  • function: Transmite bits brut through mediul physical
  • PDU: Bit (or simbol)
  • Preocupari: levels of tensiune, sincronizare, conectori
  • examples: cables Ethernet, optical fibres, radio Wi-Fi

Standard cheie: IEEE 802.3 (physical layer Ethernet)

the level 2 ‚Äì Data Link

  • function: Livrare node-la-node, detectia errorslor
  • PDU: Frame
  • Adresare: addresses MAC (48 bits)
  • examples: Ethernet, Wi-Fi, PPP

MAC Address: 00:1A:2B:3C:4D:5E ‚Äì identificator hardware

the level 4 ‚Äì Transport

communication cap-la-cap between processes

TCP ‚Äì Transmission Control Protocol

  • Orientat connection: handshake in 3 pasi
  • Fiabil: Confirmari, retransmisii
  • Ordonat: Datele are livrate in sequence
  • Control flow & congestion: Incorptimet

Folositi cand: each byte matters (HTTP, SSH, email)

UDP ‚Äì User Datagram Protocol

  • Without connection: Without handshake, without stare
  • Nefiabil: Without garantii
  • Neordonat: Pachetele can ajunge in alt ordine
  • Overhead minim: Header of 8 bytes

Folositi cand: Viteza matters (DNS, VoIP, gaming)

Socket: Combinatia (IP address, numar port) identifies unic un punct final. Numerele of port are 0‚Äì65535.

API Socket

Socketsle offers interface of progralarge between the application dvs. (the level 7) and transport layer (the level 4).

Socket: A punct final for communication. Identificat through (IP address, numar port). API allows creation connectionslor, sendsrea and receptia datalor.

Historical Note

API socket a fost introdus in BSD UNIX 4.2 (1983). A became standardul of facto for programming of network.

Acelasi API is folosit in aproape orice SO: Linux, Windows (Winsock), macOS.

types of Socket

Alegerea socket potrivit for nevoile dvs.

SOCK_STREAM (TCP)

  • Orientat connection
  • Flux of bytes fiabil, ordonat
  • Necesit connect() before of sendsre
  • Use send() / recv()
import socket
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect(("example.com", 80))

SOCK_DGRAM (UDP)

  • Without connection
  • Nefiabil, orientat messages
  • Nu requires connection
  • Use sendto() / recvfrom()
import socket
sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sock.sendto(data, ("example.com", 53))

Example Server TCP in Python

import socket

# Create socket: AF_INET = IPv4, SOCK_STREAM = TCP
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
sock.bind(("0.0.0.0", 9090))  # Leag on all interfacesle
sock.listen(32)  # Backlog of 32 connections in asteptare

while True:
    conn, addr = sock.accept()  # Block pan se connects un client
    data = conn.recv(1024)
    response = b"OK: " + data.upper()
    conn.sendall(response)
    conn.close()

Conceptii Gresite Frecvente

Clarifying common confusions

‚ùå Misconception: ‚ÄûDatele travel orizontal between levels pair."

‚úì Reality: Datele travel in jos through stiv at emitator, through mediul physical, then in sus at receptor.

‚ùå Misconception: ‚ÄûTCP guarantees delivery."

‚úì Reality: TCP guarantees delivery or notification of failure. If the network is unreachable, TCP eventually gives up.

‚ùå Misconception: ‚ÄûThe OSI model is used in real networks."

‚úì Reality: TCP/IP is what is implemented. OSI is a reference model for discussion.

‚ùå Misconception: ‚ÄûMAC addresses do not change during routing."

‚úì Reality: Source/destination MAC addresses change at each hop. IP addresses remain constant (except with NAT).
