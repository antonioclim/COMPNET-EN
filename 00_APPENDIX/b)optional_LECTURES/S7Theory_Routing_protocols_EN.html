A Single Misconfiguration, a Global Meltdown

25 April 1997 — The Day the Internet Lost Its Way

A small Internet service provider in Virginia, operating Autonomous System 7007, began advertising itself as the optimal path to nearly the entire IPv4 address space. Within minutes, routers across North America and Europe accepted these bogus announcements and redirected traffic accordingly. For roughly two hours, millions of packets converged on infrastructure wholly incapable of handling them. E-mail queues stalled, e-commerce transactions failed and network operations centres scrambled to identify the source of the chaos.

The root cause was not a hardware failure or a cyberattack. It was a routing misconfiguration — a set of incorrect announcements propagated by the very protocols designed to keep the Internet connected.

The question we shall answer today: How do routers decide where to send packets? What algorithms power this decision? And how can a single misconfigured rule cascade into a global outage?

The Problem We Cannot Yet Solve

Consider this scenario. You have three networks connected by two routers:

A host on Network A sends a packet to 10.0.3.5. Router R1 receives it. How does R1 know to forward it towards R2? We configured addresses in Week 5 — but we never told R1 that Network C exists.

This is the routing problem. Without a mechanism to populate routing tables — either manually or via a protocol — routers are blind beyond their directly connected interfaces. Today we learn how they see.

There are fundamentally two approaches: (a) manual configuration (an administrator types each route into each router) or (b) dynamic discovery (routers exchange information automatically using a protocol). Approach (a) is static routing; approach (b) is dynamic routing. Both have their place, and we shall study both today.

What Is a Router?

Definition: A router is a network device that connects two or more IP networks and makes per-packet forwarding decisions based on a routing table.

A router performs a primary function (forwarding packets between interfaces) and several secondary functions:

Primary: Packet Forwarding

Receives a packet on one interface, consults its routing table and sends the packet out through the appropriate interface towards the next hop.

Secondary Functions

Packet filtering (ACLs), Network Address Translation (recall Week 6), Quality of Service (QoS) prioritisation and tunnelling for VPNs.

Key distinction: A router operates at Layer 3 (network layer). A switch operates at Layer 2 (data-link). A router forwards based on IP addresses; a switch forwards based on MAC addresses.

Yes. Most consumer “routers” are actually multi-function devices that combine a router (L3 forwarding between WAN and LAN), a switch (L2 forwarding between LAN ports), a wireless access point and often a DHCP server. In enterprise networks, these functions are separated into dedicated devices for performance and manageability.

Routing vs Forwarding

These two terms are often confused, but they describe fundamentally different processes.

Forwarding (data plane)

The local, per-packet decision: “send this packet out via interface eth1 towards next hop 10.0.2.1.”

Happens at wire speed, for every packet. Implemented in hardware (ASICs) on modern routers.

Analogy: reading a signpost at an intersection and turning right.

Routing (control plane)

The background process that learns network topology and populates the routing table: “to reach 10.0.3.0/24, go via 10.0.2.1.”

Runs periodically or event-driven. Implemented in software.

Analogy: the cartographer who draws the map before the journey begins.

Separating data plane from control plane allows routers to forward packets at line rate even while the routing protocol converges after a topology change. It also enables Software-Defined Networking (SDN), which we encounter in Seminar 6, where the control plane is centralised and the data plane remains distributed.

What Changes at Each Hop?

As a packet traverses multiple routers, some header fields remain constant and others are rewritten at every hop.

Common misconception: Students often believe routers modify the source or destination IP address. They do not — unless NAT is configured (recall Week 6). The IP header identifies the end-to-end conversation; the MAC header identifies the next physical hop.

The Routing Table

Every router — and every host — maintains a routing table in RAM. It contains a set of entries, each mapping a destination prefix to a next hop.

A packet with destination IP 10.0.3.5 arrives at R1. R1’s routing table has entries for 10.0.3.0/24 (via 10.0.2.2) and 0.0.0.0/0 (via 10.0.1.1). Which next hop does R1 select?

A) 10.0.2.2 — the /24 entry matches and is more specific than the default
B) 10.0.1.1 — the default route always takes priority
C) The packet is dropped because two entries match
D) R1 modifies the source IP to its own address before forwarding
✔ Correct. Longest prefix match selects the /24 entry over the /0 default. The default route is a catch-all for destinations with no more specific match.
✘ Not quite. Routers use longest prefix match: among all entries whose prefix contains the destination, the one with the longest mask wins. Here, /24 beats /0. The default route is not “higher priority” — it is the least specific. And routers do not rewrite IP addresses unless NAT is configured.

Static vs Dynamic Routes

Static Routes (manual)

Configured by an administrator: ip route add 10.0.3.0/24 via 10.0.2.2

Advantages: precise control, no protocol overhead, predictable behaviour, well suited to stub networks (single exit point).

Disadvantages: does not adapt to failures. If a link goes down, the static route keeps pointing at a black hole until a human intervenes.

Dynamic Routes (via protocols)

Learned automatically: routers exchange information and calculate paths.

Advantages: scales to large topologies, adapts to link failures and topology changes, reduces administrative burden.

Disadvantages: protocol overhead (bandwidth, CPU), convergence delay, potential for routing loops during transient states.

In practice: Most enterprise and ISP networks use dynamic routing (OSPF, BGP) for the core, with static routes for stub links or default gateways. The two approaches complement rather than compete with each other.

Routing Metrics

When multiple paths exist to the same destination, the routing protocol must choose one. It does so using a metric — a numerical cost assigned to each path.

MetricUsed ByMeaning
Hop countRIPNumber of routers on the path. Simple but ignores link speed.
Cost (inverse bandwidth)OSPFReference bandwidth / link bandwidth. Prefers faster links.
CompositeEIGRPCombines bandwidth, delay, load and reliability (weighted formula).
Administrative costVariousIf multiple protocols report the same prefix, the one with the lowest administrative distance wins.

When two paths share the same metric, some protocols support Equal-Cost Multi-Path (ECMP) routing, distributing traffic across both.

A 2-hop path through two 100 Gbps links is faster than a 1-hop path through a single 10 Mbps link. But a hop-count metric picks the 1-hop path because it has fewer hops. This is exactly the kind of mistake RIP makes and OSPF avoids.

Networks from a Router’s Perspective

A router classifies every known destination into one of three categories:

Connected

Networks attached directly to the router’s own interfaces. These appear automatically when an interface is configured and brought up.

Known (remote)

Networks learned via static configuration or a dynamic routing protocol. The router has a next hop for these prefixes.

Unknown

Destinations for which no routing table entry exists. If a default route is configured, traffic goes there. Otherwise, the packet is dropped and an ICMP Destination Unreachable message is sent back (recall Week 6).

Asymmetric routing: Each router makes its forwarding decision locally and independently. There is no guarantee that the outbound and return paths are identical. This is normal behaviour — not a fault.

IGP vs EGP

The Internet is divided into Autonomous Systems (ASes) — administrative domains operated by a single organisation (an ISP, a university, a large enterprise). Routing protocols are classified by scope:

IGP — Interior Gateway Protocol

Routes within a single AS. Optimises for convergence speed, metric accuracy and operational simplicity.

Examples: RIP OSPF EIGRP IS-IS

EGP — Exterior Gateway Protocol

Routes between ASes. Optimises for policy control, scalability and path selection based on business relationships.

Example: BGP (the protocol that holds the Internet together). We mention it today for context; it is typically covered in Internet operations courses.

Today we focus on two IGPs: RIP (distance-vector) and OSPF (link-state). The AS 7007 incident from our hook? That was a BGP misconfiguration — an EGP problem with global consequences.

Enhanced Interior Gateway Routing Protocol (EIGRP) is a Cisco-proprietary protocol (later published as RFC 7868) that blends distance-vector and link-state ideas. It uses a composite metric (bandwidth, delay, reliability, load) and the DUAL algorithm for rapid convergence. In this course we focus on the two “pure” families — RIP (distance-vector) and OSPF (link-state) — because they illustrate the underlying principles most clearly. You may encounter EIGRP in Cisco-centric environments.

Distance-Vector Protocols

Routing by rumour: each router knows only what its neighbours tell it

RIP, Bellman–Ford and the count-to-infinity problem

The Distance-Vector Idea

Distance-vector routing: Each router maintains a vector (table) of distances (metrics) to every known destination. It shares this vector with its directly connected neighbours at regular intervals. No router knows the full topology.

The process works as follows:

1
Each router initialises its table with directly connected networks (distance 0).
2
Periodically, each router sends its entire routing table to all neighbours.
3
When a router receives a neighbour’s table, it checks: “Can I reach any destination more cheaply through this neighbour?” If so, update.
4
Repeat until no table changes — the network has converged.

The underlying mathematical model is the Bellman–Ford equation: d(x, dest) = min over neighbours y of { cost(x, y) + d(y, dest) }

RIP — Routing Information Protocol

RIP is the oldest dynamic routing protocol still found in textbooks. It implements the distance-vector approach with hop count as its sole metric.

The 15-hop ceiling is not a design flaw — it is a deliberate safety mechanism. RIP uses “infinity” (16) to signal unreachable destinations. In networks wider than 15 routers, RIP simply cannot function. This makes it impractical for anything larger than a small campus.

The Count-to-Infinity Problem

Distance-vector protocols have a fundamental vulnerability: when a link fails, routers can enter a loop of mutual misinformation.

Scenario: Routers A, B and C form a chain: A—B—C. Network X is connected to C. When the C—X link fails, C sets its distance to X as ∞. But before C’s update reaches B, B sends its old table to C saying “I can reach X in 2 hops (through you).” C thinks: “B can reach X in 2 hops, so I can reach it in 3!” Now B hears C’s update: “3 hops” — better than ∞, so B updates to 4. The metrics keep incrementing until they reach 16 (infinity). This is the count-to-infinity problem.

With a 30-second update interval and infinity at 16, count-to-infinity can take up to 16 × 30 = 480 seconds (8 minutes) to resolve in the worst case. During this period, packets to Network X are black-holed or looped. This is unacceptable for production networks.

Mitigating Count-to-Infinity

RIP employs several mechanisms to reduce (but not eliminate) the problem:

Split Horizon

Never advertise a route back through the interface from which it was learned. If B learned about X through C, B does not advertise X back to C.

Route Poisoning

When a route becomes unreachable, immediately advertise it with metric 16 (∞) rather than simply removing it. This explicitly tells neighbours to discard it.

Holddown Timer

After a route is poisoned, ignore new information about it for a set period. This prevents stale updates from resurrecting a dead route.

Triggered Updates

Do not wait for the next periodic update. When a route changes, send an immediate update to accelerate convergence.

Even with these mechanisms, RIP convergence remains slow compared to link-state protocols. This is the fundamental trade-off: simplicity at the cost of speed.

RIP Timers

RIP’s behaviour is governed by four timers. Understanding them is essential for predicting convergence behaviour.

Update Timer (30 s)

Every 30 seconds, the router broadcasts its entire routing table to all neighbours. This is the heartbeat of the protocol.

Invalid Timer (180 s)

If no update mentions a route for 180 seconds, it is marked as potentially invalid (metric set to 16). The route is not yet removed.

Flush Timer (240 s)

If the route remains invalid for an additional 60 seconds (total 240 s from last valid update), it is permanently removed from the table.

Holddown Timer (180 s)

After a route is poisoned, the router ignores new information about that destination for 180 seconds. This prevents routing loops from stale data.

Implication: In the worst case, RIP can take over 4 minutes to converge after a single link failure. For comparison, OSPF typically converges in under 1 second on well-tuned networks.

Bellman–Ford Algorithm

The Bellman–Ford algorithm computes shortest paths from a single source to all other nodes in a weighted graph. It tolerates negative edge weights (unlike Dijkstra) and forms the mathematical basis of distance-vector routing.

Core principle: Relax every edge repeatedly. After |V| − 1 iterations (where |V| is the number of vertices), all shortest distances are guaranteed correct — provided there are no negative-weight cycles.
1
Set dist[source] = 0 and dist[v] = ∞ for all other vertices.
2
For each iteration (1 to |V| − 1): for every edge (u, v, w), if dist[u] + w < dist[v], update dist[v].
3
After all iterations, check for negative cycles: if any edge can still be relaxed, a negative cycle exists.

Complexity: O(|V| × |E|) — slower than Dijkstra for non-negative weights, but simpler and naturally distributed.

Bellman–Ford — Worked Example

Consider a graph with vertices A, B, C, D and the following edges:

Init: dist = [A:0, B:∞, C:∞, D:∞]

Iteration 1: Relax A→B (1), A→C (4), A→D (5) ⇒ [A:0, B:1, C:4, D:5]

Iteration 2: Relax B→D (1+2=3 < 5) ⇒ [A:0, B:1, C:4, D:3]

Iteration 3: No changes — converged.

Bellman–Ford in Python

def bellman_ford( vertices: list[str], edges: list[tuple[str, str, int]], source: str ) -> dict[str, int]: # Initialise distances dist = {v: float("inf") for v in vertices} dist[source] = 0 # Relax all edges |V| - 1 times for _ in range(len(vertices) - 1): for u, v, w in edges: if dist[u] + w < dist[v]: dist[v] = dist[u] + w return dist

Usage with the graph from the previous slide:

result = bellman_ford( ["A", "B", "C", "D"], [("A", "B", 1), ("A", "C", 4), ("B", "D", 2), ("C", "D", 3), ("A", "D", 5)], "A" ) print(result) # {'A': 0, 'B': 1, 'C': 4, 'D': 3}

Knowledge Check: Distance-Vector

In RIP, a router learns that Network X is reachable through Neighbour A in 3 hops, and through Neighbour B in 2 hops. It currently has Network X in its table with metric 4 via Neighbour C. What happens?

A) The router keeps all three entries for load balancing
B) The router installs the route via B (metric 3: B’s 2 hops + 1 hop to reach B)
C) The router keeps the existing route via C because it was learned first
D) The router discards all routes because multiple paths indicate a loop
✔ Correct. The router adds 1 hop (cost to reach the advertising neighbour) to each advertised metric. B says 2 hops, so total = 3. A says 3 hops, so total = 4. The existing route via C costs 4. The route via B (cost 3) is best and replaces the existing entry. RIP does not do per-packet load balancing by default.
✘ Remember: RIP adds 1 to the advertised metric (B’s 2 becomes 3, A’s 3 becomes 4). The lowest metric wins. RIP does not load-balance across unequal paths, and “first learned” is not a tiebreaker — the metric is. Multiple paths do not indicate a loop; they simply offer route selection choices.

Link-State Protocols

Each router knows the full map and computes paths independently

OSPF, Dijkstra’s algorithm, areas and the Link-State Database

The Link-State Idea

Link-state routing takes a fundamentally different approach from distance-vector. Instead of sharing distances, each router shares its own view of its directly connected links with every other router in the network.

Distance-Vector (recall)

“I can reach Network X in 3 hops.”

Shares conclusions. The receiver trusts the neighbour’s arithmetic without knowing the topology.

Link-State

“I am Router R3. I have links to R1 (cost 10) and R4 (cost 5).”

Shares raw data. Every router builds the same complete topology map and computes routes locally using Dijkstra’s algorithm.

Link-State Database (LSDB): The complete topology map constructed from all routers’ link-state advertisements (LSAs). Every router in an OSPF area holds an identical copy of the LSDB. This is the key invariant that guarantees loop-free routing.

OSPF — Open Shortest Path First

OSPF (RFC 2328 for OSPFv2, RFC 5340 for OSPFv3) is the most widely deployed link-state IGP. It was designed to overcome the limitations of RIP in large networks.

Protocol characteristics

Uses IP protocol number 89 (not TCP or UDP). Sends hello packets via multicast (224.0.0.5 for AllSPFRouters, 224.0.0.6 for AllDRouters). Metric is cost, calculated as reference bandwidth divided by link bandwidth.

Why OSPF over RIP?

No hop-count limit. Converges in seconds (not minutes). Supports VLSM and CIDR natively. Provides authentication. Scales via area hierarchy. Uses bandwidth-aware metrics.

OSPF cost formula: cost = reference_bandwidth / interface_bandwidth. The default reference is 100 Mbps, meaning a 100 Mbps link has cost 1 and a 10 Mbps link has cost 10. Modern networks often increase the reference to 10 Gbps or 100 Gbps to differentiate between fast links.

OSPF Operation

OSPF establishes adjacencies, synchronises databases and computes routes through a well-defined sequence:

1
Hello: Routers discover neighbours by sending hello packets every 10 seconds (on broadcast networks). The hello contains the router ID, area ID, hello/dead timers and the list of known neighbours.
2
Database synchronisation: Once neighbours form an adjacency, they exchange Database Description (DBD) packets, request missing LSAs and synchronise their LSDBs to become identical.
3
DR/BDR election: On multi-access segments (e.g. Ethernet), a Designated Router (DR) and Backup DR are elected to reduce the number of adjacencies from O(n²) to O(n). All routers synchronise with the DR, not with each other.
4
SPF calculation: Each router runs Dijkstra’s algorithm on its LSDB to compute the shortest-path tree rooted at itself. This produces the routing table entries.

The new router sends Hello packets on its interfaces. Existing OSPF neighbours detect it and begin adjacency formation. The neighbours exchange their LSDBs (Database Description packets, then Link-State Requests). Once synchronised, the new router has a complete topology map and can run SPF. The entire process takes seconds on a well-configured network — far faster than waiting for RIP’s 30-second update cycle.

OSPF Areas

In large networks, a flat LSDB would become enormous and SPF recalculations expensive. OSPF solves this with a two-level hierarchy of areas.

Area 0 is the backbone — all other areas must connect to it. Area Border Routers (ABRs) summarise routes between areas, reducing the size of each area’s LSDB and the cost of SPF calculations.

If Area 0 is split, inter-area routing breaks because all traffic between non-backbone areas must transit Area 0. OSPF requires a contiguous backbone. If physical contiguity is impossible, administrators use virtual links (RFC 2328, §15) to logically bridge the gap through a transit area.

Dijkstra’s Algorithm

Dijkstra’s algorithm (Edsger Dijkstra, 1956) computes the shortest path from a single source to all other nodes in a graph with non-negative edge weights. OSPF runs Dijkstra on the LSDB to build its routing table.

Core principle: Greedily select the unvisited node with the smallest tentative distance, mark it as visited and relax all its outgoing edges.
1
Set dist[source] = 0, all others to ∞. Create a priority queue.
2
Extract the node u with the smallest tentative distance from the queue.
3
For each neighbour v of u: if dist[u] + weight(u, v) < dist[v], update dist[v].
4
Repeat steps 2–3 until the queue is empty.

Complexity: O((|V| + |E|) log |V|) with a binary heap. Faster than Bellman–Ford for non-negative weights.

Dijkstra — Worked Example

Same graph as before. Source: A. Let us trace Dijkstra step by step.

Both algorithms produce the same result for this graph. The difference: Dijkstra visits each node once (greedy); Bellman–Ford relaxes all edges repeatedly.

Dijkstra in Python

import heapq def dijkstra( graph: dict[str, list[tuple[str, int]]], source: str ) -> dict[str, int]: # Initialise distances dist = {v: float("inf") for v in graph} dist[source] = 0 pq = [(0, source)] while pq: d, u = heapq.heappop(pq) if d > dist[u]: continue for v, w in graph[u]: if dist[u] + w < dist[v]: dist[v] = dist[u] + w heapq.heappush(pq, (dist[v], v)) return dist

Usage:

g = { "A": [("B", 1), ("C", 4), ("D", 5)], "B": [("D", 2)], "C": [("D", 3)], "D": [], } print(dijkstra(g, "A")) # {'A': 0, 'B': 1, 'C': 4, 'D': 3}

Predict: Which Algorithm, Which Protocol?

Before we compare RIP and OSPF side by side, test your mental model.

A network engineer must run shortest-path computation on a router with 200 nodes in its topology database. Which statement is correct?

A) The router uses Bellman–Ford because it is simpler and faster on large graphs

✔ Correct. OSPF routers run Dijkstra on their LSDB. Because link costs (derived from bandwidth) are always non-negative, Dijkstra’s greedy approach works and is asymptotically faster than Bellman–Ford. RIP is the protocol that conceptually uses Bellman–Ford, not OSPF.
✘ OSPF uses Dijkstra (not Bellman–Ford). Bellman–Ford underpins distance-vector protocols like RIP. OSPF link costs are always non-negative (typically 108 / bandwidth), so Dijkstra’s assumption holds and it converges faster. Routers do not mix algorithms or use hop count in OSPF.

Knowledge Check: Link-State vs Distance-Vector

An OSPF network has three paths between routers A and D. Path 1: two 1 Gbps links. Path 2: one 100 Mbps link. Path 3: four 10 Gbps links. Reference bandwidth is 10 Gbps. Which path does OSPF select?

A) Path 2 — fewest hops (1 hop)
B) Path 1 — good balance of hops and speed
C) Path 3 — lowest total cost (4 × 1 = 4)
D) OSPF cannot decide and drops the packet
✔ Correct. With reference 10 Gbps: Path 1 cost = 2 × (10000/1000) = 20. Path 2 cost = 1 × (10000/100) = 100. Path 3 cost = 4 × (10000/10000) = 4. Path 3 has the lowest total cost despite having the most hops.
✘ Remember the OSPF cost formula: cost = reference_bw / link_bw, summed over all links on the path. Path 1: 2×10 = 20. Path 2: 1×100 = 100. Path 3: 4×1 = 4. OSPF does not count hops — it sums costs. RIP would pick Path 2 (1 hop), but OSPF correctly identifies that Path 3’s fast links make it the best choice.

What If the Direct Link Fails?

Suppose the R1—R2 link goes down in our triangle topology.

With static routes only

R1’s route to R2’s network points at a dead interface. Packets are dropped silently until an administrator manually reconfigures the route to go via R3. There is no automatic recovery.

With a dynamic protocol (e.g. OSPF)

R1 detects the adjacency loss within seconds (dead timer expiry). It removes the failed link from its LSDB, floods an updated LSA to R3, recalculates SPF and installs a new route: reach R2 via R3. Convergence takes under 1 second on a well-tuned network.

RIP would also eventually converge, but it takes much longer. The invalid timer (180 s) must expire before the route is marked unreachable. With triggered updates and split horizon, convergence improves somewhat — but it is still measured in tens of seconds to minutes, compared to OSPF’s sub-second recovery.

AS 7007 Revisited

We can now understand the hook from Slide 2. The AS 7007 incident was a BGP route leak: an EGP problem, not an IGP problem. But it illustrates the same principle we studied today at a global scale.

What went wrong

AS 7007’s router was misconfigured to redistribute its full internal routing table into BGP, announcing itself as the best path to thousands of prefixes it did not actually serve. Neighbouring ASes accepted these announcements because BGP (at the time) lacked adequate filtering.

What we now understand

Routing protocols propagate information — correct or incorrect — through the network. A single bad entry can cascade because every router trusts its protocol’s output. This is why route filtering, authentication and prefix validation (RPKI, developed years later) are essential.

Narrative closure: The AS 7007 event became a catalyst for BGP security improvements. It demonstrated that the power of routing protocols — their ability to propagate reachability information automatically — is also their greatest vulnerability when misconfigured.

Synthesis Question

A company has 50 routers in a single site, all connected via Ethernet. An engineer proposes using RIP for routing. Which is the strongest argument against this choice?

A) RIP uses too much bandwidth for updates
B) RIP does not support Ethernet networks
C) With 50 routers, some paths will exceed 15 hops, making portions of the network unreachable
D) RIP cannot operate without a designated router
✔ Correct. The 15-hop ceiling is the fundamental scalability limit of RIP. In a network of 50 routers, the diameter (longest shortest path) can easily exceed 15 hops, rendering portions of the network unreachable. OSPF, with its area hierarchy and no hop limit, is the appropriate choice here.
✘ While RIP’s periodic broadcasts do consume some bandwidth, the primary issue with 50 routers is the 15-hop limit. Paths longer than 15 hops are treated as unreachable (metric 16 = infinity). RIP works fine on Ethernet, and the DR/BDR concept belongs to OSPF, not RIP. The hop limit is the decisive factor.

You will capture packets traversing routed paths, build a Python packet filter and implement a port scanner. Understanding routing tables helps you predict where captured packets came from and why they took a particular path.

Week 8 — Transport Layer

TCP and UDP sit directly above the network layer. TCP’s retransmission logic assumes that IP routing eventually delivers packets. When routes change mid-connection, TCP’s sequence numbers and ACKs keep the conversation intact.

Seminar 5–6 — Already Done

You configured static routes in Mininet and explored SDN topologies. Today’s lecture explains why those static routes worked and what happens when you replace them with dynamic protocols.

Week 13 — Security

Route hijacking (like AS 7007), prefix filtering and RPKI are network security topics. Understanding routing protocols is a prerequisite for understanding routing-based attacks.

Key Takeaways

1. Forwarding ≠ Routing

Forwarding is the per-packet data-plane decision. Routing is the background control-plane process that populates the forwarding table.

2. Routing table = longest prefix match

The most specific matching entry wins. The default route (0.0.0.0/0) is the catch-all of last resort.

3. Distance-vector: simple but slow

RIP shares distances with neighbours. Hop count ignores bandwidth. Count-to-infinity is a real risk. Convergence takes minutes.

4. Link-state: powerful but complex

OSPF shares link information with everyone. Each router computes routes locally via Dijkstra. Converges in sub-seconds. Scales via areas.

5. Metrics determine path selection. RIP counts hops; OSPF measures cost (inverse bandwidth). The same topology can produce different optimal paths depending on the metric. Choose the protocol that measures what matters.

If you can articulate these three points, you have grasped the core: (1) OSPF converges in sub-seconds versus minutes for RIP; (2) OSPF uses bandwidth-based cost, not hop count, so it picks genuinely faster paths; (3) OSPF scales via areas, whereas RIP collapses beyond 15 hops. For small, simple topologies RIP suffices; for anything else, OSPF is the standard IGP.
