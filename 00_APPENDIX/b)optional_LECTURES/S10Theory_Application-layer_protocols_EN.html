“It Works in Postman but Not in the Browser”

A developer’s worst Friday afternoon

A junior developer at a fintech startup in Bucharest has built a REST API. Every endpoint returns the correct JSON when tested with Postman or curl. The frontend team integrates the API into their React application. They deploy on Friday at 16:00.

Within minutes, the dashboard shows nothing. The browser console is filled with red errors: Access to fetch at 'https://api.example.com' has been blocked by CORS policy. The API is running perfectly — yet the browser refuses to use it.

The Symptom
The API responds correctly to every tool except the browser. The HTTP response is identical. Something outside HTTP is blocking the request.
The Question
Why would a browser reject a perfectly valid HTTP response? What mechanism exists above HTTP that decides whether your code may use a response?

By the end of this lecture, you will understand exactly why this happens — and how to prevent it.

The Protocol Hiding in Plain Sight

You have already built applications with HTTP — React frontends, Express backends, REST APIs with an ORM. But have you ever stopped to ask what the protocol actually does when your fetch() call fires?

What You Know
HTTP has methods (GET, POST), returns status codes (200, 404) and carries JSON. You have used it many times in your Web Technologies course.
What You Assume
HTTP “just works” over the Internet. Sending a request is like calling a function — you send input and get output.
What Actually Happens
DNS resolution, TCP handshake, TLS negotiation, header parsing, content negotiation, caching validation, cookie management and browser security policies — all before your code sees the response.
Today’s shift in perspective

This lecture does not teach you how to build an API — you already know that. It teaches you what the protocol does beneath the framework, so that when things break, you can diagnose them.

Where We Are — The Journey So Far

Recall from Weeks 8–9

TCP provides reliable, ordered byte streams between two endpoints. TLS adds confidentiality and authentication on top. The session layer manages long-lived dialogues. Today we build on all three: HTTP runs over TCP (or QUIC), may use TLS for security and maintains application-level sessions via cookies and tokens.

Quick Check — Are We Ready?

Before we proceed, let us verify that the prerequisite concepts are solid.

An HTTP request to https://example.com must complete several steps before the first byte of the response arrives. Which sequence is correct?

A) HTTP request → TCP handshake → DNS lookup → TLS handshake
B) DNS lookup → TCP handshake → TLS handshake → HTTP request
C) TLS handshake → DNS lookup → HTTP request → TCP handshake
D) DNS lookup → HTTP request → TCP handshake → TLS handshake
Correct. The browser must first resolve the hostname (DNS), then establish a TCP connection (SYN-ACK), negotiate encryption (TLS) and only then send the HTTP request. Each step depends on the previous one — you cannot encrypt a connection that does not yet exist.
Not quite. HTTP cannot be sent before a TCP connection exists, and TCP requires knowing the server’s IP address, which comes from DNS. The dependencies form a strict chain: DNS → TCP → TLS → HTTP.
TLS requires a TCP connection underneath it, and TCP requires an IP address from DNS. You cannot negotiate encryption on a connection that has not been established yet.
HTTP cannot be sent before the TCP and TLS layers are established. The HTTP request is the last step, not an intermediate one. Think of it as building upwards through the stack: each layer depends on the one below.

The Problem: TCP Works, but the Web Doesn’t

You built a TCP server in Seminar 2. You can send any bytes to any port. Why, then, do browsers refuse to simply connect and exchange data?

What TCP gives you
A reliable byte stream between two endpoints. You send bytes; they arrive in order. TCP knows nothing about “requests”, “headers” or “status codes”.
What the web requires
Structured messages with metadata. Authentication. Content negotiation. Caching. Security policies. Connection reuse. Compression. State management. None of this exists at the transport layer.
The gap that HTTP fills

HTTP defines a contract between clients and servers: a shared language for requesting resources, describing content, managing state and handling errors. Without it, every web application would need to invent its own protocol from scratch — and none of them would interoperate.

The application layer is where meaning enters the network stack. TCP delivers bytes; HTTP gives them purpose.

What Makes Application Protocols Special?

Application-layer protocols operate closest to the user. They define the semantics — the meaning — of communication.

Application-Layer Protocol (OSI Layer 7)

A set of rules governing how processes on different hosts exchange structured messages to accomplish a task. Unlike transport protocols (which move bytes), application protocols interpret those bytes as requests, responses, commands or notifications.

SECTION 01

HTTP Fundamentals

The request-response protocol that powers the modern web — from header structure to the hidden cost of every byte.

HTTP in the Network Architecture

HTTP (Hypertext Transfer Protocol) is an application-layer, request–response protocol that runs over TCP (or, since HTTP/3, over QUIC). It was specified originally in RFC 1945 (HTTP/1.0, 1996) and subsequently refined in RFCs 7230–7235 (HTTP/1.1, 2014) and RFC 7540 (HTTP/2, 2015).

LayerProtocolRole for HTTP
7 – ApplicationHTTP / HTTPSRequest-response semantics
6 – PresentationTLS 1.2 / 1.3Encryption for HTTPS
5 – SessionCookies, tokensApplication-level state
4 – TransportTCP (or QUIC)Reliable delivery
3 – NetworkIPRouting to destination
Common misconception

HTTP does not “just work over the Internet”. A single HTTP request involves DNS resolution, one or more TCP handshakes, possibly a TLS negotiation and routing through multiple autonomous systems. Most web performance problems are protocol problems, not framework problems.

HTTP is stateless by design: each request is independent. The protocol carries no memory of previous exchanges. Applications layer state on top via cookies and tokens — a direct link to the session-layer concepts from Week 9.

Anatomy of an HTTP Request

Every HTTP request consists of three sections: the request line, a set of headers and an optional body.

GET /api/users/42 HTTP/1.1 Host: api.example.com User-Agent: Mozilla/5.0 (Windows NT 10.0) Accept: application/json Authorization: Bearer eyJhbGciOiJIUzI1NiIs... Cache-Control: no-cache
Request Line
METHOD path HTTP/version
Specifies the action and the target resource. The path is relative to the host.
Headers
Name: value pairs conveying metadata — authentication, content type, caching directives, accepted encodings.
Body
Optional payload, typically present in POST, PUT and PATCH requests. Interpreted only through the Content-Type header.

Anatomy of an HTTP Response

The response mirrors the request structure: a status line, headers and an optional body.

HTTP/1.1 200 OK Content-Type: application/json; charset=utf-8 Content-Length: 128 Cache-Control: max-age=3600 X-Request-Id: a3f8c2d1-e9b7-4f2a {"id": 42, "name": "Ada Lovelace", "role": "engineer"}

Content-Type tells the client how to interpret the body (JSON, HTML, binary). Content-Length declares the body size in bytes — essential for TCP framing. Cache-Control instructs the browser (and any intermediate proxies) to store this response for 3600 seconds. X-Request-Id is a custom header for correlating logs across distributed systems.

Headers are not decoration — they are instructions that control caching, security, encoding and routing behaviour across every component in the request path.

HTTP Methods: Semantics, Not Convention

HTTP methods carry formal semantic guarantees defined in RFC 7231. These guarantees are not suggestions — proxies, caches and load balancers rely on them.

MethodPurposeSafe?Idempotent?Cacheable?
GETRetrieve a resourceYesYesYes
HEADGET without bodyYesYesYes
POSTCreate / processNoNoRarely
PUTReplace entire resourceNoYesNo
PATCHPartial modificationNoNoNo
DELETERemove a resourceNoYesNo
Why does this table matter?

Safe means the method does not alter server state — a proxy may pre-fetch safe resources. Idempotent means repeating the request produces the same final state — load balancers and retry logic depend on this. Cacheable means intermediaries may store and reuse the response without contacting the server.

Why Idempotency Matters in Practice

Scenario: The duplicate payment

A user clicks “Pay Now”. The request times out after 10 seconds. The client retries automatically. The server processes the payment twice. The customer is charged double.

What went wrong? The payment endpoint used POST /pay — a non-idempotent method. The retry created a second transaction.

Option 1 — Idempotency key: The client generates a unique token (e.g. UUID) and sends it with the request: Idempotency-Key: 550e8400-.... The server stores it and rejects duplicates.

Option 2 — PUT semantics: Design the endpoint as PUT /payments/{id} where the client chooses the resource identifier. Repeating the same PUT with the same body produces the same result.

Idempotency is not a theoretical nicety — it is the foundation of reliable distributed systems. Network timeouts, load balancer retries and browser refresh all generate duplicate requests. Your API must handle them.

Status Codes: Fine-Grained Semantics

Status codes are grouped into five classes. Each class carries a specific meaning that clients, proxies and browsers interpret automatically.

1xx — Informational
100 Continue, 101 Switching Protocols (used in WebSocket upgrade)
2xx — Success
200 OK, 201 Created, 204 No Content
3xx — Redirection
301 Moved Permanently, 304 Not Modified (caching)
4xx — Client Error
400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 405 Method Not Allowed, 422 Unprocessable Entity
5xx — Server Error
500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout
304 Not Modified

A response without a body that tells the browser: “Use the copy you already have.” This single status code can eliminate 90% of redundant traffic for static assets.

Checkpoint — Status Code Diagnostics

A developer sends a request with a valid JWT token to DELETE /api/users/99. The server returns 403 Forbidden. What is the most likely explanation?

A) The JWT token has expired or is malformed.
B) The resource /api/users/99 does not exist.
C) The user is authenticated but lacks permission to delete this resource.
D) The server does not support the DELETE method.
Correct. 403 Forbidden means the server understood the request and recognised the caller, but the caller does not have authorisation for this specific action. The distinction: 401 = who are you? (authentication missing), 403 = I know who you are, but you may not do this (authorisation denied).
An expired or malformed token would produce 401 Unauthorized, not 403. The 401 code signals that authentication is missing or invalid. A 403 presupposes that authentication succeeded but the caller lacks permission.
A non-existent resource would produce 404 Not Found. The server returns 403 specifically to say “the resource exists, you are authenticated, but you lack permission.”
If the server did not support DELETE on this endpoint, it would return 405 Method Not Allowed, typically with an Allow header listing the permitted methods.

Content Negotiation: Content-Type vs Accept

HTTP enables clients and servers to agree on the format and encoding of exchanged data through header-based negotiation.

Content-Type — “What I am sending”

The sender declares the format of the body:

Content-Type: application/json; charset=utf-8

Without this header, the receiver has no reliable way to interpret the bytes in the body.

Accept — “What I can receive”

The client requests a preferred format:

Accept: application/json, text/html;q=0.9

The q parameter (quality) expresses preference: 1.0 is highest, 0 means “not acceptable”.

A prolific source of bugs

Sending a JSON body with Content-Type: text/plain (or omitting the header entirely) is one of the most common API integration errors. The server will either reject the request (415 Unsupported Media Type) or misparse the body. Similarly, sending Accept: text/xml when the server only produces JSON yields 406 Not Acceptable.

HTTP Caching: A Mechanism, Not Magic

Caching reduces latency and bandwidth by storing responses at multiple points: the browser, intermediate proxies and CDN (Content Delivery Network) edge nodes.

Freshness — “Is this still valid?”
Cache-Control: max-age=3600 tells the browser this response is fresh for one hour. During that window, no request is sent to the server at all.
Validation — “Has it changed?”
When the cached copy expires, the browser sends a conditional request: If-Modified-Since: Thu, 20 Feb 2026 10:00:00 GMT. The server responds either with a fresh copy (200) or a bodyless 304 Not Modified.

Cookies: Application State over a Stateless Protocol

HTTP is stateless by design — each request is independent. Applications need state (sessions, preferences, shopping carts), so the protocol provides a mechanism: cookies, defined in RFC 6265.

# Server sets a cookie Set-Cookie: session_id=abc123; Path=/; Secure; HttpOnly; SameSite=Strict; Max-Age=3600
# Browser sends it back automatically Cookie: session_id=abc123
Security Flags

Secure — transmit only over HTTPS.

HttpOnly — invisible to JavaScript (prevents XSS theft).

SameSite — controls cross-site sending (CSRF defence).

Session layer connection

Cookies implement at the application level what the session layer (L5) models abstractly: persistent identity across multiple exchanges. This is the bridge between the stateless protocol and the stateful application.

CORS: The Browser Security Model

Remember the Friday afternoon disaster from Slide 2? CORS is the mechanism behind it.

Cross-Origin Resource Sharing (CORS)

A browser-enforced policy that restricts web pages from making requests to a different origin (scheme + host + port) than the one that served the page. It is not an HTTP limitation — it is a browser security decision.

Same-origin (allowed)
https://app.example.com requests https://app.example.com/api — same scheme, host and port. The browser sends the request normally.
Cross-origin (blocked unless server opts in)
https://app.example.com requests https://api.example.com — different host. The browser sends a preflight OPTIONS request to ask permission.

Why Postman works and browsers do not

Postman, curl and Python’s requests library are not browsers. They do not enforce the same-origin policy. CORS is a browser feature, not a server feature. The HTTP response is identical in both cases — the browser simply refuses to let your JavaScript read it unless the server explicitly grants permission via Access-Control-Allow-Origin headers.

Checkpoint — CORS Understanding

A React application at https://frontend.io calls https://api.backend.io/data. The server does not include any Access-Control-* headers. The browser shows a CORS error. What actually happened at the network level?

A) The browser did not send the request. The server never received it.
B) The server rejected the request because it came from a different origin.
C) The server received and processed the request. The browser received the response but refused to expose it to JavaScript.
D) The firewall between the two servers blocked the cross-origin traffic.
Correct. For simple requests (GET without custom headers), the browser sends the request normally. The server processes it and returns a response. The browser then checks the response headers for Access-Control-Allow-Origin. If the header is missing or does not match, the browser hides the response from JavaScript — but the server already did the work. This is a critical distinction: CORS is client-side enforcement, not server-side rejection.
For simple GET requests, the browser does send the request. It only blocks before sending for non-simple requests that trigger a preflight OPTIONS. In this case, the request reaches the server.
The server knows nothing about CORS unless it has been explicitly programmed to check the Origin header. CORS is enforced by the browser, not the server. The server happily processes the request and sends a response.
CORS is not a network-level mechanism. Firewalls operate at layers 3–4 (IP, TCP); they know nothing about HTTP origin headers. This error occurs entirely within the browser.
SECTION 02

HTTP Evolution and WebSocket

From sequential requests to multiplexed streams — and what happens when request-response is no longer enough.

From HTTP/1.1 to HTTP/2

HTTP/1.1 (RFC 7230, 1997) introduced persistent connections but retained a fundamental constraint: head-of-line blocking. HTTP/2 (RFC 7540, 2015) resolved this with multiplexing.

The key insight

HTTP/2 changes the transport framing (how bytes flow on the wire) without changing HTTP semantics (methods, headers, status codes). Your application code does not change — only the underlying connection behaviour improves.

HTTP/3 and QUIC: The UDP Revolution

HTTP/2 solved HTTP-level head-of-line blocking but exposed a TCP-level problem: if a single TCP packet is lost, all streams on that connection stall until retransmission completes. HTTP/3 (RFC 9114, 2022) replaces TCP with QUIC, a UDP-based transport with built-in TLS 1.3.

FeatureHTTP/2 over TCPHTTP/3 over QUIC
TransportTCPQUIC (UDP-based)
EncryptionSeparate TLSTLS 1.3 integrated
HandshakeTCP + TLS (2–3 RTT)1 RTT (0-RTT resumption)
HoL blockingAt TCP levelPer-stream only
Connection migrationBreaks on IP changeSurvives Wi-Fi ↔ mobile
Real-world impact

Google reported that QUIC reduced search latency by 8% globally and video rebuffering by 18% on YouTube. The largest gains appear on high-latency, lossy networks — precisely the conditions mobile users face.

QUIC runs over UDP, and many corporate firewalls and middleboxes block or deprioritise UDP traffic. Additionally, QUIC’s encryption makes deep packet inspection impossible, which conflicts with some enterprise security policies. Adoption is growing but not universal.

Checkpoint — HTTP Versions

Consider the following scenario.

A web page loads 30 small images. Under HTTP/1.1 with a single connection, the browser must wait for each response before sending the next request. Under HTTP/2, all 30 requests can be sent immediately over the same connection. Why?

A) HTTP/2 uses UDP instead of TCP, which allows parallel delivery.
B) HTTP/2 introduces multiplexed streams within a single TCP connection, eliminating HTTP-level head-of-line blocking.
C) HTTP/2 opens 30 separate TCP connections in parallel.
D) HTTP/2 compresses images before sending, so they arrive faster.
Correct. HTTP/2 divides communication into numbered streams. Frames from different streams can be interleaved on a single TCP connection, so the browser does not need to wait for one response to finish before sending the next request. The underlying TCP connection is shared.
HTTP/2 still uses TCP. It is HTTP/3 that uses QUIC (which runs over UDP). The multiplexing in HTTP/2 is achieved through a binary framing layer, not a transport change.
Opening 30 TCP connections is actually the HTTP/1.1 workaround (browsers typically open 6 connections per host). HTTP/2 explicitly reduces this to one or two connections by multiplexing streams within them.
HTTP/2 compresses headers (via HPACK), not bodies. Image compression is unrelated to the protocol version and is handled by the image format itself (JPEG, PNG, WebP).

WebSocket: When HTTP Is No Longer Enough

HTTP follows a strict request–response pattern: the client initiates, the server responds. The server cannot push data to the client unprompted. For real-time applications (chat, live dashboards, gaming), this limitation is fundamental.

WebSocket Protocol (RFC 6455)

A persistent, full-duplex communication channel established via an HTTP upgrade handshake. After the handshake, both sides can send frames at any time without waiting for a request.

# Client sends upgrade request GET /chat HTTP/1.1 Upgrade: websocket Connection: Upgrade Sec-WebSocket-Key: dGhlIHNh... # Server confirms HTTP/1.1 101 Switching Protocols Upgrade: websocket Connection: Upgrade
After the handshake
The connection is no longer HTTP. It becomes a bidirectional, frame-based protocol over the same TCP connection. Both client and server may send data at any moment.
Polling vs WebSocket

Polling: Client asks “any updates?” every N seconds — wasteful, high latency.

WebSocket: Server sends updates as they happen — no wasted requests, real-time delivery.

Trade-off: WebSocket keeps a TCP connection open permanently, consuming server resources per connected client.

SECTION 03

REST Architecture

An architectural style, not a technology — modelling a domain as resources with HTTP semantics.

REST: An Architectural Style, Not a Technology

REST (Representational State Transfer) was introduced by Roy Fielding in his doctoral dissertation at UC Irvine (2000). It is a set of architectural constraints for distributed hypermedia systems — not a protocol, not a standard and not a library.

The six constraints of REST

1. Client–server separation — concerns are divided; the UI and data storage evolve independently. 2. Statelessness — each request contains all information needed to process it; the server stores no client context. 3. Cacheability — responses must declare whether they are cacheable. 4. Layered system — intermediaries (proxies, gateways, CDNs) may exist transparently. 5. Uniform interface — resources are identified by URIs; manipulation happens through representations with standard methods. 6. Code on demand (optional) — servers may send executable code (e.g. JavaScript) to extend client functionality.

The most common misconception in the industry

REST is not “a JSON API over HTTP”. An API that uses JSON and HTTP but violates statelessness or ignores cacheability is not RESTful, regardless of its URL structure. Conversely, a perfectly RESTful API could serve XML, HTML or even plain text.

The Richardson Maturity Model

Leonard Richardson proposed a model with four levels to classify how well an API uses HTTP semantics. Most production APIs reach Level 2; Level 3 remains rare.

Each level adds a new dimension of HTTP usage. The jump from Level 0 to Level 2 is where the most practical benefit lies: correct use of methods enables caching, makes load balancing safer and reduces debugging time.

Levels 0 and 1 in Practice

Level 0 — RPC over HTTP

# Everything goes to one endpoint POST /api HTTP/1.1 Content-Type: application/json { "action": "getUser", "userId": 42 } POST /api HTTP/1.1 { "action": "deleteUser", "userId": 42 }

HTTP is merely a transport tunnel. The “action” field inside the body replaces HTTP methods. Caching is impossible because every request is POST to the same URL.

Level 1 — Resources with URIs

# Each resource has its own URI POST /api/users/42 HTTP/1.1 { "action": "get" } POST /api/users/42 HTTP/1.1 { "action": "delete" }

Resources are addressable, but POST is still used for everything. The “action” field encodes what should be an HTTP method. Still not cacheable.

Levels 2 and 3 — Proper REST

Level 2 — HTTP verbs + status codes

GET /api/users/42 HTTP/1.1 # 200 OK + JSON body (cacheable!) DELETE /api/users/42 HTTP/1.1 # 204 No Content (idempotent) POST /api/users HTTP/1.1 {"name": "Ada"} # 201 Created + Location header

GET responses can be cached by browsers and CDNs. DELETE is idempotent — safe for automatic retries. POST returns 201 with the URL of the new resource.

Level 3 — HATEOAS

{ "id": 42, "name": "Ada Lovelace", "_links": { "self": { "href": "/api/users/42" }, "projects": { "href": "/api/users/42/projects" }, "delete": { "href": "/api/users/42", "method": "DELETE" } } }

The response tells the client what it can do next. The client navigates the API by following links, not by hardcoding URLs. This is the principle behind the web itself: hypertext.

Checkpoint — REST in Practice

An API endpoint POST /api/users/42/activate changes the user’s status to “active”. At which Richardson maturity level does this endpoint operate?

A) Level 2 — because it uses a URI for the resource and POST for the action.
B) Level 1 — it has a resource URI but encodes the action in the URL path instead of using HTTP methods.
C) Level 3 — because it targets a specific sub-resource of the user.
D) Level 0 — because POST is used instead of GET.
Correct. The verb “activate” in the URL is a strong indicator of RPC-style thinking encoded in a resource path. At Level 2, you would model this as PATCH /api/users/42 with a body like {"status": "active"}, using the HTTP method (PATCH) to express the action and the body to express the change. The URL /activate is an action, not a resource.
Level 2 requires using HTTP methods to express operations. Here, the action “activate” is baked into the URL path, which means the API is using the URL as a verb rather than letting the HTTP method carry that information. This is characteristic of Level 1.
Level 3 (HATEOAS) requires the server to include hypermedia links in responses, guiding the client to available actions. The URL structure alone does not determine Level 3.
Level 0 uses a single endpoint for all operations (e.g. POST /api). Here, the API does address a specific resource URI, placing it above Level 0.

SOAP: The Enterprise Alternative

SOAP (Simple Object Access Protocol) predates REST and follows a fundamentally different philosophy: a strict, contract-driven protocol with XML messaging and formal service descriptions.

<!-- A SOAP request envelope --> <soap:Envelope xmlns:soap="...xmlsoap.org/..."> <soap:Header> <!-- Auth, routing metadata --> </soap:Header> <soap:Body> <GetUser> <UserId>42</UserId> </GetUser> </soap:Body> </soap:Envelope>
Key characteristics

WSDL — a machine-readable contract (Web Services Description Language) describing every operation, its parameters and return types.

WS-* extensions — enterprise features: WS-Security, WS-Transaction, WS-ReliableMessaging.

Transport-independent — can run over HTTP, SMTP or even JMS.

When SOAP makes sense

Banking systems, healthcare (HL7 FHIR interfaces) and government services often require formal contracts, digital signatures and guaranteed delivery — features that SOAP provides natively and REST must implement manually.

REST vs SOAP — Different Problems, Different Tools

The pragmatic view

REST dominates modern web development because it aligns with HTTP’s native semantics and works well with browsers, CDNs and JavaScript. SOAP persists where formal contracts and enterprise-grade security are non-negotiable. They solve different problems; neither is universally superior.

Returning to Friday Afternoon

The mystery, resolved

Remember the fintech developer from Slide 2? The React frontend at https://dashboard.fintech.ro called the API at https://api.fintech.ro. Different subdomain → different origin → cross-origin request. The server was missing three headers:

# The fix: add these headers to every response Access-Control-Allow-Origin: https://dashboard.fintech.ro Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS Access-Control-Allow-Headers: Authorization, Content-Type
What you now understand

CORS is a browser policy, not an HTTP feature. The server must explicitly opt in to cross-origin access. Postman bypasses this because it is not a browser.

The fix took three lines of configuration. The diagnosis required understanding how HTTP, browsers and security policies interact — precisely the knowledge this lecture provides.

The broader lesson

HTTP is not a black box. It is a precisely defined protocol with documented semantics. When your framework hides these details, everything works. When something breaks, you need to see through the abstraction to the protocol underneath.

Synthesis Challenge

A mobile banking application uses HTTP/2 to communicate with its backend API via HTTPS. The server returns Cache-Control: no-store on all responses. A user submits a payment using POST /api/payments. The network connection drops mid-request. The client retries automatically. Which combination of HTTP features prevents a double payment?

A) HTTP/2 multiplexing ensures the retry reaches the same stream.
B) The Cache-Control: no-store header prevents the response from being reused.
C) An Idempotency-Key header sent by the client allows the server to detect and reject the duplicate.
D) HTTPS encryption ensures that retransmitted packets are identical and the server can deduplicate them.
Correct. Since POST is not idempotent, an automatic retry can create a second payment. The solution is an application-level idempotency mechanism: the client generates a unique key (UUID) and includes it with each request. The server stores the key after processing and rejects any subsequent request bearing the same key. This combines transport-layer understanding (retries happen), HTTP semantics (POST is non-idempotent) and application-level design (idempotency keys).
HTTP/2 multiplexing organises multiple requests within a connection, but a dropped connection destroys all streams. The retry starts a new TCP + TLS connection; the server has no memory of the previous stream.
Cache-Control: no-store prevents caching but has no effect on duplicate request handling. It tells intermediaries not to store the response — it says nothing about duplicate submissions.
HTTPS encrypts the transport channel but does not provide application-level deduplication. TCP retransmission handles lost segments within a connection, but a new connection (after the original drops) is a completely independent request from the server’s perspective.
