The Forged Letter That Nobody Questioned

A Scenario You Can Reproduce in Five Minutes

Open a terminal. Connect to any SMTP server on port 25. Type MAIL FROM:<rector@ase.ro>. The server replies 250 OK. You have just set the envelope sender to the university rector — and the server accepted it without a single challenge. The original Simple Mail Transfer Protocol (SMTP), defined in 1982, includes no mechanism to verify the identity of the sender.

Real Consequence

In 2020, security researchers demonstrated that over 80% of government domains lacked proper email authentication. Attackers could forge official emails indistinguishable from genuine ones.

The Question

How does an email system designed with zero authentication still carry 340 billion messages per day? What protections have been layered on top — and where do the gaps remain?

Can You Trust the “From:” Field?

Consider the email below. It arrived in your inbox. Is it genuine?

From: IT Security <security@ase.ro> To: student@stud.ase.ro Subject: Urgent: Password Reset Required Date: Mon, 10 Mar 2026 09:15:00 +0200 Your account will be locked in 24 hours. Click here to reset your password immediately.
The Uncomfortable Truth

You cannot tell from these headers alone. The From: header is set freely by the sender — it is display information, not verified identity. By the end of this lecture, you will know exactly which records to check (SPF, DKIM, DMARC) and how to inspect the full header chain to distinguish authentic messages from forgeries.

What We Have Built So Far

Email protocols sit at the application layer — the same layer as HTTP (Week 10) and FTP, DNS and SSH (Week 11). Recall these foundations:

TCP Connections

Email protocols run over TCP (recall Week 8). SMTP, POP3 and IMAP all require reliable, ordered delivery.

DNS Resolution

When you send email to bob@example.com, your mail server queries DNS for MX records (recall Week 11) to locate the recipient’s server.

Client–Server Model

Email follows the same request–response pattern as HTTP. Text commands from the client; numeric status codes from the server.

Week 11 → Week 12: Last week we saw how DNS translates names to addresses and how SSH secures remote sessions. Today we examine the protocols that power the most persistent application on the internet: electronic mail.

Quick Check: Are You Ready?

Your mail server needs to deliver a message to alice@example.com. Which DNS record type does it query?
A) A record — resolves a hostname to an IPv4 address
B) CNAME record — creates an alias for another domain
C) MX record — specifies the mail exchanger for a domain
D) PTR record — maps an IP to a hostname
Exactly right. MX (Mail Exchanger) records tell the sending server which host accepts email for the target domain. Without MX records, email delivery would require hard-coded server addresses — an approach that cannot scale.
Not quite. A records resolve hostnames to IPs, CNAME creates aliases, PTR performs reverse lookups. For email routing, the sending MTA queries the MX record of the recipient’s domain (recall Week 11: DNS record types).

The Problem: One Message, Five Protocols

You open Gmail on your phone, compose a message to a colleague who uses Outlook on a laptop, and press Send. That single action triggers at least five distinct protocol interactions.

Can You Trace the Full Path?

The message must be submitted to a server, routed across the internet, delivered to a remote mailbox, formatted with attachments and HTML, and finally retrieved by the recipient’s client. Each step uses a different protocol — each designed in a different decade under different trust assumptions.

With what we know so far (TCP sockets, DNS, HTTP), we could build a raw connection to port 25 and type commands. But we cannot yet explain why there are separate protocols for sending and reading, how attachments survive the ASCII-only constraint of SMTP, or what prevents anyone from impersonating any sender.

Section 01

Email Architecture

The postal system of the internet: agents, protocols and the journey of a message.

The Four Agents of the Email System

Email architecture mirrors a traditional postal service. Four specialised components divide the work.

AgentFull NameRolePostal Analogy
MUAMail User AgentInterface for composing and readingPen, paper and letterbox
MSAMail Submission AgentAccepts outgoing mail from MUA (port 587)Post office counter
MTAMail Transfer AgentRoutes messages between servers via SMTPSorting centres and trucks
MDAMail Delivery AgentPlaces message into recipient mailboxPostman at your door
Why Separate Agents?

Separation of concerns. The MUA never touches routing; the MTA never renders HTML. Each component can be implemented independently — Postfix for MTA, Dovecot for MDA, Thunderbird for MUA.

The Journey of an Email

Sending (SMTP)

MUA → MSA (port 587, authenticated) → MTA queries DNS MX → recipient MTA (port 25) → MDA → mailbox.

Retrieving (POP3 / IMAP)

Recipient MUA connects to the mailbox via POP3 (110/995) or IMAP (143/993) to download or synchronise.

Envelope vs Message Headers

The single most misunderstood concept in email: routing information and display information are completely separate.

SMTP Envelope (Routing)

MAIL FROM:<alice@sender.com> RCPT TO:<bob@receiver.com>

Set during SMTP session. Invisible to end user. Determines where bounces go.

Message Headers (Display)

From: Alice <alice@sender.com> To: Bob <bob@receiver.com> Subject: Meeting tomorrow

Embedded in the message. Shown by the email client. Can be freely set by the sender.

The Spoofing Gap

Because envelope and headers can differ, a malicious sender can set MAIL FROM:<attacker@evil.com> in the envelope while the header says From: ceo@yourcompany.com. The recipient sees only the header. This is exactly how phishing works — and why SPF, DKIM and DMARC were invented.

Section 02

SMTP

Simple Mail Transfer Protocol: the workhorse that moves billions of messages daily.

SMTP — Simple Mail Transfer Protocol

RFC 5321 (2008) — supersedes RFC 2821 and the original RFC 821 (1982)

SMTP is the standard protocol for transferring email between servers and for submitting email from a client to its outgoing server. Text-based, command–response, over TCP.

Port 25

Server-to-server relay. Often blocked by ISPs for residential connections.

Port 587

Message submission. Requires authentication. Correct port for MUA → MSA.

Port 465

SMTPS (implicit TLS). Re-standardised by RFC 8314 in 2018.

SMTP Commands

CommandParametersPurposeReply
EHLOclient hostnameExtended greeting; requests extensions250
MAIL FROM<address>Specifies envelope sender (return-path)250
RCPT TO<address>Specifies envelope recipient (repeatable)250
DATA—Begins message body; ends with a lone dot354, then 250
QUIT—Terminates the session221
Historical Note

The original HELO (RFC 821, 1982) had no extension support. EHLO (RFC 1869) added negotiation for STARTTLS, AUTH and 8BITMIME. Modern servers should always use EHLO.

SMTP Response Codes — Explorer

SMTP replies use a three-digit structure. The first digit indicates the category. Type a code to see its meaning.

A Complete SMTP Session

S: 220 mail.example.com ESMTP Postfix C: EHLO client.domain.com S: 250-mail.example.com S: 250-STARTTLS S: 250 AUTH PLAIN LOGIN C: MAIL FROM:<alice@sender.com> S: 250 OK C: RCPT TO:<bob@receiver.com> S: 250 OK C: DATA S: 354 End data with <CR><LF>.<CR><LF> C: From: Alice <alice@sender.com> C: To: Bob <bob@receiver.com> C: Subject: Meeting notes C: C: Hello Bob, find the notes attached. C: . S: 250 OK: queued as ABC123 C: QUIT S: 221 Bye

Sending Email with Python

The smtplib module wraps SMTP commands into a clean, Pythonic API.

import smtplib from email.mime.text import MIMEText # Build the message object msg = MIMEText( "Hello Bob, here are the notes.", "plain", "utf-8" ) msg["Subject"] = "Meeting notes" msg["From"] = "alice@sender.com" msg["To"] = "bob@receiver.com" # Connect, authenticate and send with smtplib.SMTP( "smtp.sender.com", 587 ) as server: server.starttls() server.login("alice", "password") server.send_message(msg) print("Message sent.")

Knowledge Check: SMTP

A colleague claims: “I use SMTP to read my emails every morning.” Is this correct?
A) Yes — SMTP handles both sending and reading
B) No — SMTP is a transfer protocol; reading requires POP3 or IMAP
C) Partially — SMTP can read if the server supports EHLO extensions
D) It depends on whether the server uses port 25 or 587
Correct. SMTP is exclusively a transfer protocol. It pushes messages from sender to recipient server. To retrieve messages, the client must use POP3 or IMAP.
SMTP stands for Simple Mail Transfer Protocol — it moves messages between servers. It has no commands for listing, fetching or managing a mailbox. That is the job of POP3 or IMAP.
Section 03

POP3 & IMAP

Two philosophies for mailbox access: download-and-delete vs server-side synchronisation.

POP3 — Post Office Protocol version 3

RFC 1939 (1996)

POP3 is a simple download protocol. Design philosophy: download-and-delete. Retrieve messages to the local client, optionally remove them from the server.

CommandPurposeResponse
USER nameIdentifies the mailbox owner+OK
PASS secretAuthenticates the user+OK logged in
STATReturns message count and total size+OK n octets
LISTLists all messages with sizes+OK (multi-line)
RETR nRetrieves message number n+OK (content)
DELE nMarks message n for deletion+OK marked
QUITCommits deletions and closes+OK bye

POP3 in Action

POP3 sessions are short-lived: connect, authenticate, download, disconnect. Predict the flow before revealing:

S: +OK POP3 server ready C: USER alice S: +OK C: PASS secret123 S: +OK alice has 3 messages (12480 octets) C: STAT S: +OK 3 12480 C: RETR 1 S: +OK 4520 octets (message content follows...) C: DELE 1 S: +OK message 1 deleted C: QUIT S: +OK POP3 server signing off (2 left)

Strengths

Simple implementation. Offline after download. Minimal server storage. Ports: 110 (plain), 995 (TLS).

Weaknesses

No sync across devices. No folders. No partial download. Local failure = lost messages.

IMAP — Internet Message Access Protocol

RFC 9051 (2021) — IMAP4rev2, supersedes RFC 3501 (2003)

IMAP provides full remote mailbox management. Messages remain on the server. Clients synchronise state rather than downloading entire messages. Multi-device access is native.

CommandPurposeReply
LOGINAuthenticate with credentialsOK LOGIN completed
SELECTOpen a mailbox folderOK [READ-WRITE]
FETCHRetrieve message data (headers, body, parts)* FETCH ...
SEARCHFind messages matching criteria* SEARCH 1 4 7
STOREModify message flagsOK STORE completed
EXPUNGEPermanently remove \Deleted messagesOK EXPUNGE completed

IMAP Session States and Flags

IMAP tracks message state with flags, synchronised across every connected client:

\Seen

Message has been read.

\Answered

User replied to this message.

\Flagged

Marked important / starred.

\Deleted

Marked for removal (EXPUNGE).

\Draft

Work-in-progress composition.

\Recent

Arrived since last session.

POP3 vs IMAP — Side by Side

AspectPOP3IMAP
StorageDownload to local clientKeep on server
Multi-device syncNo — each device has its own stateYes — all devices share one view
FoldersINBOX onlyFull hierarchy
FlagsNot supported\Seen, \Answered, \Flagged, etc.
Server searchNoYes (SEARCH command)
Industry reality: IMAP is the default for modern email. Gmail, Outlook.com and nearly all enterprise providers recommend IMAP. POP3 persists for legacy systems or low-bandwidth environments.

Knowledge Check: Mailbox Access

A user reads an email on their phone. On their laptop, the same email still shows as unread. Which protocol is their phone using?
A) POP3 — downloaded without updating server state
B) IMAP — synchronises read status across devices
C) SMTP — the mail transfer was incomplete
D) HTTP — the webmail interface cached old state
Correct. POP3 downloads but does not synchronise state. The laptop still sees the original unread state.
The symptom is lack of sync. IMAP would have set \Seen on the server. The protocol that does not propagate read-state is POP3.
Section 04

MIME

How email outgrew plain ASCII: attachments, HTML bodies and multimedia content.

MIME — Multipurpose Internet Mail Extensions

The 7-Bit Problem

Original SMTP (RFC 821, 1982) allowed only 7-bit US-ASCII in the message body. No accented characters, no images, no attachments. MIME (RFCs 2045–2049, 1996) solved this with content types and transfer encodings that wrap arbitrary data into ASCII-safe formats.

What MIME Enables

Non-ASCII text (UTF-8), file attachments, HTML-formatted bodies, multipart messages combining text and files.

Three Key Headers

MIME-Version: 1.0 Content-Type: text/html; charset=utf-8 Content-Transfer-Encoding: base64

Content Types and Multipart Structure

Common Content-Types

text/plain — plain text
text/html — HTML body
multipart/mixed — body + attachments
multipart/alternative — two formats
application/pdf — PDF file
image/png — PNG image

Multipart Boundary

Content-Type: multipart/mixed; boundary="----=_Boundary" ------=_Boundary Content-Type: text/plain; charset=utf-8 Document attached below. ------=_Boundary Content-Type: application/pdf Content-Disposition: attachment; filename="report.pdf" Content-Transfer-Encoding: base64 JVBERi0xLjQKJcOkw7zD... ------=_Boundary--
Boundary Rules

Each part starts with --boundary. The final boundary ends with --boundary--. The boundary string must not appear inside the content.

Section 05

Email Security

SPF, DKIM and DMARC: the triad that defends against spoofing and forgery.

Email Was Designed in an Era of Trust

Original SMTP assumed everyone was who they claimed to be. Four decades later, the consequences persist.

Spoofing

Any SMTP client can set any From: header. Sender identity is unverifiable without external mechanisms.

Phishing

Spoofed emails impersonating banks or colleagues remain the primary credential theft vector.

Interception

Without TLS, email travels as plaintext between servers — readable by any intermediary.

Spam

Approximately 45% of global email traffic is unsolicited bulk mail.

Defence stack: SPF (who may send), DKIM (was the message altered?), DMARC (what policy on failure).

SPF — Sender Policy Framework

RFC 7208 (2014)

SPF lets a domain owner publish a DNS TXT record listing which IPs are authorised to send email for that domain.

example.com. IN TXT "v=spf1 ip4:192.0.2.0/24 include:_spf.google.com -all"

Mechanisms

ip4: match range • include: delegate • mx match MX servers • -all hard fail rest.

Verdicts

pass — authorised. fail — not. softfail — probably not. neutral — no statement.

DKIM — DomainKeys Identified Mail

RFC 6376 (2011)

DKIM adds a cryptographic signature to email headers. The sending server signs selected headers and body hash with a private key; the public key is published in DNS.

Signing (Sender)

Hash headers + body → Sign with private key → Add DKIM-Signature header → Publish public key in DNS.

Verification (Receiver)

Extract d=, s= → Query DNS for public key → Recompute hash → Verify signature.

DKIM-Signature: v=1; a=rsa-sha256; d=example.com; s=selector1; h=from:to:subject:date; b=dzdVyOfAKCdLXJOc...

DMARC — Domain-based Message Authentication

RFC 7489 (2015)

DMARC ties SPF and DKIM together by checking alignment: does the From: header domain match the domain verified by SPF or DKIM?

p=none

Monitor only. Collect reports.

p=quarantine

Move failures to spam.

p=reject

Reject outright. Strongest.

_dmarc.example.com. IN TXT "v=DMARC1; p=quarantine; rua=mailto:dmarc-reports@example.com"

Knowledge Check: Email Security

You receive email from hr@yourcompany.com. SPF passes. DKIM passes. DMARC fails. Most likely cause?
A) TLS encryption prevented the receiver from reading the message
B) The sender’s DNS has no MX record
C) The domain verified by SPF/DKIM does not match the From: header domain
D) The message exceeded the maximum SMTP size
Precisely. DMARC checks alignment: the authenticated domain must match the From: header domain. If they differ, DMARC fails even though SPF and DKIM passed individually.
DMARC’s unique role is alignment: the authenticated domain must match the From: header domain. When they differ, DMARC fails regardless of individual SPF/DKIM results.

Revisiting the Forged Letter

At the start we showed that anyone can set MAIL FROM:<rector@ase.ro>. Now we have the full picture.

SPF Check

Attacker’s IP not in ase.ro SPF record → SPF fail.

DKIM Check

Attacker lacks ase.ro private key → DKIM fail.

DMARC Policy

If p=reject, receiving server discards the forgery.

Answer: Trust the From: field only if the receiver checks SPF, DKIM and DMARC and the sender publishes correct records. Both sides must participate.
Section 06

Modern Email and Application

WebMail, Python implementations and debugging delivery failures.

WebMail Architecture

WebMail (Gmail, Outlook.com, Roundcube) is not a new protocol — it is a web frontend translating HTTP into SMTP and IMAP on the server side.

How It Works

Browser ↔ HTTPS ↔ WebMail server ↔ IMAP (read) + SMTP (send). No email protocol logic in the browser.

Advantages

No client installation. Any browser. Integrated contacts, calendars and filtering.

Modern Trend: Proprietary APIs

Gmail API and Microsoft Graph use REST/GraphQL instead of raw IMAP, offering faster sync and richer features (labels, AI sorting). The trade-off is vendor lock-in.

Reading Email with Python

The imaplib module provides low-level IMAP access. Compare with smtplib from slide 16.

import imaplib # Connect over TLS mail = imaplib.IMAP4_SSL( "imap.example.com", 993 ) # Authenticate mail.login( "alice@example.com", "password" ) # Select the inbox folder mail.select("INBOX") # Search for unread messages status, msg_ids = mail.search( None, "UNSEEN" ) # Fetch the first unread message for mid in msg_ids[0].split()[:1]: status, data = mail.fetch( mid, "(RFC822)" ) print(data[0][1].decode("utf-8")) mail.logout()

Key Takeaways

Architecture

Email flows through MUA → MSA → MTA → MDA → Mailbox. DNS MX records direct messages.

Envelope vs Headers

SMTP envelope = routing. Message headers = display. They can differ — root cause of spoofing.

Three Protocols, Three Roles

SMTP transfers. POP3 downloads. IMAP synchronises. MIME enables rich content.

Security Triad

SPF checks IP. DKIM verifies integrity. DMARC enforces alignment.
