The Day Half the Internet Went Dark

21 October 2016

At 07:10 EDT, the DNS provider Dyn began receiving an unprecedented flood of traffic — over 1.2 Tbit/s from roughly 100,000 compromised IoT devices. Within minutes, DNS resolution for major websites stopped working. Twitter, Netflix, Reddit, GitHub, Spotify and dozens of other services became unreachable for millions of users across North America and Europe.

The Paradox

None of these websites were hacked. Their servers were running perfectly. The problem was that nobody could find them — because the system that translates names to addresses had been overwhelmed.

Three Questions for Today

By the end of this lecture, you will be able to answer:

  • Why did attacking a single DNS provider disable so many unrelated services?
  • How does the protocol that "finds" websites actually work — and where are its weaknesses?
  • How do FTP and SSH — two protocols from the same application layer — solve entirely different problems with radically different architectures?

What We Know — and What We Cannot Yet Explain

Recall from Week 10

Last week we studied HTTP, the request–response protocol that powers the Web. HTTP lets a client fetch resources from a server using methods such as GET and POST, over TCP port 80 (or 443 with TLS).

HTTP excels at document retrieval. But the application layer handles far more than web pages. Consider three scenarios that HTTP alone cannot solve:

Scenario A — File Transfer

You need to upload 500 files to a server, resume interrupted transfers and list remote directories. HTTP can transfer files, but has no native directory listing or session persistence.

Scenario B — Name Resolution

Before HTTP can even begin, your browser must discover the server's IP address. Who provides this mapping — and what happens if they fail?

Scenario C — Remote Administration

You need to execute commands on a server 3,000 km away, transfer configuration files and create a secure tunnel — all encrypted. HTTP was never designed for this.

SECTION 01

FTP — File Transfer Protocol

A dual-connection architecture for file transfers between heterogeneous systems — and a cautionary tale about protocol design in the age of firewalls.

What is FTP?

Definition

File Transfer Protocol (FTP) is an application-layer protocol for transferring files between a client and server over TCP, using separate connections for control commands and data transfer.

Key Characteristics

  • Operates over TCP (reliable delivery)
  • Uses two connections: control and data
  • Supports authentication and anonymous access
  • ASCII commands, binary or text file transfers

Historical Context

FTP was first specified in RFC 114 (1971), making it one of the oldest Internet protocols. The modern specification is RFC 959 (1985) — older than the World Wide Web itself.

Security Warning

Classic FTP transmits credentials in plaintext. Anyone capturing traffic between client and server can read usernames and passwords directly. Modern deployments should use FTPS or SFTP.

FTP Dual-Connection Architecture

Unlike most protocols you have studied, FTP uses two separate TCP connections simultaneously:

Why Two Connections?

Separation enables out-of-band signalling — you can abort a large file transfer (with ABOR) without tearing down the session. It also allows directory listings and transfers to occur on different channels. This design, however, creates complications with firewalls and NAT.

FTP Control Commands

The control connection uses ASCII text commands terminated with CR LF:

CommandFunction
USERSpecify username
PASSSpecify password
PWDPrint working directory
LISTList directory contents
RETRRetrieve (download) a file
STORStore (upload) a file
PASVEnter passive mode
QUITEnd the session

Response Codes

2xx — Success

220: Service ready • 230: Logged in

3xx — Intermediate

331: Username OK, need password

4xx/5xx — Errors

530: Not logged in • 550: File unavailable

Active vs Passive Mode

The data connection can be established in two ways. Think first: if a client sits behind a NAT gateway, which mode will fail — and why?

Active Mode (PORT)

  1. Client opens a local port and listens
  2. Client sends PORT ip,ip,ip,ip,p1,p2
  3. Server initiates data connection from port 20 to client

The Problem

The server tries to connect inbound to the client — but the client's firewall blocks unsolicited incoming connections.

Passive Mode (PASV)

  1. Client sends PASV
  2. Server responds with IP and an ephemeral port
  3. Client initiates the data connection outbound

Advantage

Both connections are outbound from the client — firewalls and NAT gateways allow this naturally.

Predict: FTP Through a NAT Gateway

Scenario

A student in a university dormitory (behind a NAT gateway) connects to an FTP server at ftp.example.com using active mode. The control connection on port 21 is established successfully. The student issues LIST to see files. What happens next?

Result: The Transfer Hangs

In active mode, the server attempts to open a new inbound TCP connection to the student's machine on the port announced via the PORT command. The NAT gateway has no port-forwarding rule for this port. The SYN packet from the server is silently dropped. The student sees the command hang indefinitely — with no error message, because the control channel is fine.

Switching to PASV (passive mode) solves the problem immediately: the client initiates both connections outbound, which NAT handles transparently.

  • Binary messages, not ASCII
Recommended

Common Misconception

SFTP is not "FTP running over SSH." It is a completely separate file transfer protocol defined as an SSH subsystem (draft-ietf-secsh-filexfer). It uses binary-encoded messages over a single channel — there are no ASCII commands, no PORT/PASV negotiation and no dual-connection headaches.

Knowledge Check: FTP

A colleague tells you: "We should switch from FTP to SFTP — it's just the secure version of the same protocol." What is wrong with this statement?

Nothing — SFTP is indeed FTP with encryption added on top
SFTP is a completely different protocol (SSH subsystem), not an extension of FTP
SFTP and FTP use the same port (21) but with different encryption
The statement is correct for FTPS but wrong for SFTP — they are the same thing
✓ Correct. SFTP is defined as an SSH subsystem and uses binary messages over a single encrypted channel on port 22. It shares no protocol machinery with FTP. FTPS, on the other hand, is FTP extended with TLS — but even FTPS inherits all of FTP's dual-connection complexity.
✗ This is a widespread misconception. The names are confusingly similar, but SFTP and FTP are architecturally unrelated. SFTP runs over SSH (port 22), uses binary messages and requires only a single connection. FTP uses ASCII commands over a persistent control channel (port 21) plus separate data connections.
SECTION 02

DNS — Domain Name System

The hierarchical, distributed naming system that makes the Internet human-readable — and the invisible single point of failure that brought down half the web in 2016.

The Problem DNS Solves

Imagine the Alternative

Without DNS, you would need to memorise 142.250.180.206 for Google, 157.240.1.35 for Facebook, 140.82.121.4 for GitHub... and each address could change without notice.

Definition

Domain Name System (DNS) is a hierarchical, distributed database that translates human-readable domain names into IP addresses (and vice versa).

Key properties: distributed (no single point of failure by design), hierarchical (tree structure from root to leaf) and cached (multi-level caching for performance).

Protocol Details

  • Primary transport: UDP port 53
  • TCP for zone transfers and large responses (> 512 bytes)
  • Specified in RFC 1034/1035 (1987)

Why UDP?

DNS queries are typically small (< 512 bytes) and need low latency. A UDP round-trip completes in one request–response exchange. If the response is truncated (the TC flag is set), the client retries over TCP.

The DNS Hierarchy

Root Servers

13 logical servers (a.root-servers.net through m.root-servers.net), replicated globally via anycast. They know which servers are authoritative for each TLD.

TLD Servers

Managed by registries (Verisign for .com, ROTLD for .ro). They know the authoritative nameservers for each domain registered under that TLD.

DNS Resolution: Step by Step

Iterative vs Recursive

The recursive resolver does the heavy lifting — querying root, TLD and authoritative servers in sequence. The client's stub resolver simply sends one query to the recursive resolver and waits for the final answer. This is why attacking the recursive resolver (as in the Dyn incident) has such catastrophic effects.

Why MX Records Matter

Next week (Week 12) we study email protocols. MX records are the mechanism by which DNS directs email to the correct mail server — connecting DNS directly to SMTP routing.

DNS Packet Structure (RFC 1035)

QDCOUNT 1 Questions

Key Flags

  • QR (1 bit): 0 = query, 1 = response
  • RD (1 bit): Recursion Desired
  • RA (1 bit): Recursion Available
  • RCODE (4 bits): 0 = OK, 3 = NXDOMAIN
# QNAME encoding for "www.google.com" 03 77 77 77 # length 3 + "www" 06 67 6f 6f 67 6c 65 # length 6 + "google" 03 63 6f 6d # length 3 + "com" 00 # null terminator

TTL and Caching Strategy

Time To Live (TTL)

Every DNS response includes a TTL value (in seconds) that specifies how long the record may be cached before the resolver must query again.

DNS Security: Threats and Defences

DNS Spoofing

An attacker injects false responses before the legitimate server replies. Exploits UDP's lack of built-in authentication.

DNS Hijacking

A malicious resolver returns wrong IP addresses. Sometimes done by ISPs for "helpful" error pages or by governments for censorship.

DNSSEC (RFC 4033–4035)

Cryptographic signatures on DNS records form a chain of trust from the root zone down. Validates data integrity and authenticity.

DoH / DoT

DNS over HTTPS (port 443) and DNS over TLS (port 853) encrypt queries to prevent eavesdropping and tampering on the wire.

Why is DNSSEC not universal?

Key management complexity, increased packet sizes, performance overhead and, frankly, lack of commercial incentive. Approximately 90% of TLDs support DNSSEC, but only around 30% of .com domains are actually signed.

Knowledge Check: DNS

Your colleague changed the A record for app.example.com to a new server IP 10 minutes ago, but clients still reach the old server. What is the most likely explanation?

The DNS server crashed and is not serving the new record
The old record's TTL has not yet expired in resolver caches
DNSSEC is blocking the change from propagating
The client's /etc/hosts file overrides DNS
✓ Correct. DNS changes are not instantaneous. Every resolver that cached the old record will continue serving it until the TTL expires. If the previous TTL was set to, say, 3600 seconds (1 hour), it may take up to an hour before all resolvers fetch the new record. Use dig +trace to bypass caches and verify the authoritative answer directly.
✗ While /etc/hosts can override DNS and DNSSEC can cause validation failures, the overwhelmingly common reason for delayed DNS propagation is TTL-based caching. Resolvers that already have the old answer cached will not query again until the TTL expires. This is exactly the trade-off we explored with the TTL slider.
SECTION 03

SSH — Secure Shell

Encrypted remote access, cryptographic authentication and secure tunnelling — all multiplexed over a single TCP connection.

What is SSH?

Definition

Secure Shell (SSH) is a cryptographic network protocol for secure data communication, remote command execution and network services between computers, running over TCP port 22.

Origin Story

In 1995, a password-sniffing attack on the University of Helsinki network compromised thousands of credentials. Researcher Tatu Ylönen created SSH as a direct response — a drop-in replacement for Telnet, rsh and rlogin that encrypted everything by default. The current standard is SSH-2 (RFC 4251–4254).

What SSH Provides

  • Authentication: password, public key, certificates
  • Confidentiality: AES-256, ChaCha20 encryption
  • Integrity: HMAC verification on every packet
  • Port forwarding: secure tunnels for arbitrary traffic

Protocol Details

  • Default port: TCP 22
  • Specification: RFC 4251–4254
  • Current version: SSH-2 (SSH-1 is deprecated)

SSH Protocol Architecture

Transport Layer

Negotiates encryption algorithms, performs Diffie-Hellman key exchange and verifies the server's identity via its host key. All subsequent traffic is encrypted.

Connection Layer

Multiplexes multiple independent logical channels over the encrypted transport — enabling simultaneous shell sessions, file transfers and port forwarding.

SSH Key Exchange

First Connection Warning

The message "The authenticity of host can't be established" appears when connecting for the first time. The client stores the server's host key in ~/.ssh/known_hosts. If the key changes on a subsequent connection, SSH warns of a potential man-in-the-middle attack — verify with the administrator before updating.

SSH Authentication Methods

Public Key Authentication

Client proves ownership of a private key by signing a challenge. Most secure method; recommended for all production use.

$ ssh-keygen -t ed25519

Password Authentication

Traditional password transmitted over the encrypted channel. Vulnerable to brute-force attacks; often disabled on production servers.

Keyboard-Interactive

Server sends arbitrary prompts to the client. Enables two-factor authentication, TOTP codes and challenge-response schemes.

Certificate Authentication

SSH certificates signed by a trusted CA eliminate the need for managing known_hosts and authorized_keys at scale.

Distributing Your Public Key

Copy your public key to a server: ssh-copy-id user@server — this appends ~/.ssh/id_ed25519.pub to the server's ~/.ssh/authorized_keys file.

SSH Channel Multiplexing

A single SSH connection can carry multiple independent channels simultaneously:

Port Forwarding: Local (-L)

Access a remote service as if it were running locally:

Use Case

Access a PostgreSQL database that only the bastion host can reach. After the tunnel is active, connect with psql -h localhost on your machine.

Syntax

-L [bind:]local_port:dest_host:dest_port

Traffic to local port is forwarded through SSH to dest_host:dest_port from the server's perspective.

Port Forwarding: Remote (-R)

Expose a local service to the remote network:

Use Case

Share your local development server with external testers or receive webhooks without deploying. Functions similarly to tools like ngrok.

Security Note

Requires GatewayPorts yes in the server's sshd_config to bind on public interfaces rather than just loopback.

Port Forwarding: Dynamic (-D) SOCKS Proxy

Creates a local SOCKS5 proxy that routes all traffic through the SSH server:

$ ssh -D 1080 user@proxy-server # Configure browser SOCKS5 proxy: # Host: localhost Port: 1080

Applications

  • Access internal corporate networks remotely
  • Privacy on untrusted Wi-Fi networks
  • Bypass geo-restrictions (the exit point is the server)

SSH as a Control Plane

SSH is not merely a tool for typing commands on a remote terminal. In modern infrastructure, it functions as a general-purpose control protocol — the secure transport layer over which automation systems operate.

Tools Built on SSH

  • Ansible — agentless configuration management; executes modules over SSH
  • Fabric / Invoke — Python libraries for remote command execution via Paramiko
  • scp / rsync — file synchronisation using SSH as transport
  • Git (over SSH) — git push to GitHub uses SSH authentication

The Pattern

SSH + JSON (or YAML) has become a de facto standard for lightweight infrastructure control: read a configuration file, iterate over hosts, SSH into each one, execute commands, collect output. This is essentially what Ansible does.

Knowledge Check: SSH

You need to access a PostgreSQL database (port 5432) that is only reachable from a bastion host. Which SSH command creates the correct tunnel?

ssh -L 5432:db-server:5432 user@bastion
ssh -R 5432:db-server:5432 user@bastion
ssh -D 5432 user@bastion
ssh -p 5432 user@db-server
✓ Correct. -L creates a local forward: traffic sent to localhost:5432 on your machine is tunnelled through the bastion and forwarded to db-server:5432 on the internal network. You then connect with psql -h localhost.
✗ Remember the mnemonic: -L = Local port listens (access a remote service locally), -R = Remote port listens (expose a local service remotely), -D = Dynamic SOCKS proxy (route all traffic). The -p flag merely sets the SSH port, not a tunnel.

Architectural Lesson

These three protocols illustrate three fundamentally different design approaches at the same layer: FTP's dual-connection model (clean separation, but fragile with NAT), DNS's hierarchical delegation (scalable, but dependent on caching) and SSH's layered encrypted multiplexing (versatile, but heavier). Understanding these trade-offs prepares you to evaluate the design of any application-layer protocol.

Returning to the Dyn Attack

Now We Can Explain It

The Mirai botnet directed 1.2 Tbit/s of traffic at Dyn, a major recursive DNS resolver provider. Because millions of clients depended on Dyn to translate domain names into IP addresses, the attack did not need to touch Twitter, Netflix or GitHub at all.

With the resolver overwhelmed, stub resolvers received no answers. Browsers displayed "DNS_PROBE_FINISHED_NXDOMAIN" — the sites appeared to not exist, even though their servers were fully operational.

The Structural Weakness

DNS is distributed by design, but operational concentration re-introduced a single point of failure. When one provider handles resolution for thousands of domains, it becomes an attractive target.

Mitigation Strategies

  • Use multiple DNS providers (e.g. both Cloudflare and Route 53)
  • Employ anycast to distribute load geographically
  • Set appropriate TTL values so cached records survive short outages
  • Deploy DNSSEC to prevent spoofing during recovery

Common Pitfalls and Debugging

FTP: Passive Port Range Blocked

Server's PASV port range blocked by firewall. Configure both the FTP server's passive port range and the corresponding firewall rules.

DNS: Stale Cache

Changed a record but clients see the old IP? The previous TTL has not expired. Use dig +trace to bypass all caches and query the authoritative server directly.

SSH: Host Key Changed

The warning "REMOTE HOST IDENTIFICATION HAS CHANGED" could indicate a server reinstall — or a man-in-the-middle attack. Verify with the administrator before updating known_hosts.

SSH: Permission Denied

Check: ~/.ssh permissions (700), key file (600), authorized_keys (644) and that the user is listed in AllowUsers in sshd_config.

Essential Debugging Tools

FTP Debugging

# Verbose FTP client $ ftp -d host # Wireshark filter ftp || ftp-data

DNS Debugging

# Full resolution trace $ dig +trace example.com # Query a specific server $ dig @8.8.8.8 example.com A

SSH Debugging

# Maximum verbosity $ ssh -vvv user@host # Test authentication only $ ssh -T git@github.com

Key Takeaways

FTP

  • Dual connections: control (21) + data
  • Passive mode for NAT traversal
  • Use SFTP (not FTP) in practice
  • Design lesson: separation of concerns

DNS

  • Hierarchical: root → TLD → authoritative
  • UDP port 53 (TCP for large responses)
  • TTL balances freshness vs performance
  • DNSSEC and DoH/DoT for security

SSH

  • Three layers: transport, auth, connection
  • Public key auth over password
  • -L (local), -R (remote), -D (SOCKS)
  • Also a control plane (Ansible, Git)

The Unifying Perspective

These three protocols demonstrate different architectural responses to the same layer's challenges: FTP separates concerns via dual connections, DNS delegates authority through a tree and SSH multiplexes everything through a single encrypted pipe. Each design reflects the constraints and priorities of its era — and each has lessons for modern system design.

Synthesis Question

A company's website became unreachable for 45 minutes, but their server logs show zero downtime during this period. The engineering team discovered that their DNS provider experienced an outage. Why did the outage persist for exactly 45 minutes after the provider recovered?

DNS servers take 45 minutes to restart after a crash
TCP connections to DNS servers have a 45-minute timeout
Negative responses (NXDOMAIN/SERVFAIL) were cached by resolvers with a TTL that had not yet expired
The company's firewall blocked DNS traffic for 45 minutes
✓ Correct. During the outage, resolvers received error responses (SERVFAIL or NXDOMAIN) and cached them according to the SOA record's minimum TTL or the resolver's negative caching policy. Even after the DNS provider recovered, these cached negative answers continued to be served until the negative cache TTL expired. This is why DNS TTL configuration matters for both positive and negative responses.
✗ The key is negative caching. When DNS resolution fails, resolvers do not simply forget — they cache the failure for a period defined by the SOA minimum TTL. After recovery, these stale "this domain does not exist" answers persist in caches until the negative TTL expires.
