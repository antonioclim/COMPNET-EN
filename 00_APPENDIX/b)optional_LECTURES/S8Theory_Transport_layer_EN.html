The Streaming Paradox

It is Friday evening. You are watching a live Champions League match online. In minute 87, the score is 1–1. The striker receives the ball in the penalty area… and your stream freezes for three seconds. When it resumes, the ball is in the net—but you missed the goal.

Meanwhile, your flatmate is downloading a 2 GB dataset for a machine-learning project. Not a single byte is lost, not a single byte arrives out of order—but the transfer takes far longer than expected.

Two applications, same network, same wires—yet radically different behaviour. One sacrifices reliability for speed; the other sacrifices speed for reliability. Who makes this choice? And how?

We shall answer this precisely by the end of today’s lecture—and you will be able to predict which transport protocol any given application should use.

Why the Transport Layer Matters

In Week 7 we saw how routing protocols deliver packets across networks. But routing only gets a packet to a machine. Your laptop runs dozens of processes simultaneously—a browser, an email client, a chat application, a file download. The transport layer answers a question that IP cannot:

Which process on this machine should receive this particular data?

What IP provides

Host-to-host delivery—getting a packet from source machine to destination machine across arbitrary network topologies.

What transport adds

Process-to-process delivery—directing data to the correct application, optionally with reliability, ordering and flow control.

Where We Are in the Course

We have climbed from physical signals (W4) through IP addressing (W5–W6) and routing (W7). Today we reach the transport layer—the boundary where the network begins serving individual applications.

Quick Recall: What Happens After Routing?

Recall from Week 7: a router examines the destination IP address, consults its routing table and forwards the packet towards the next hop. Eventually the packet arrives at the destination host. But then what?

Checkpoint: A packet arrives at host 192.168.1.50. Three applications are running: a web browser, an SSH client and a DNS resolver. How does the OS know which application should receive this packet?

The OS reads the application name from the IP header
The OS uses the destination port number in the transport-layer header
The OS delivers the packet to all three applications simultaneously
The application that sent the request keeps polling the network interface
Correct. The transport layer introduces port numbers—16-bit identifiers that the OS uses to demultiplex incoming data to the correct process. The IP header carries only source and destination addresses; it has no concept of which application is communicating. This is the problem the transport layer solves.

The Problem IP Cannot Solve

IP addresses identify machines, not applications. Without an additional mechanism, the OS cannot distinguish data destined for the browser from data destined for the email client. The transport layer introduces exactly this mechanism: the port.

The Transport Layer in the TCP/IP Stack

The transport layer bridges “get a packet to that machine” (IP’s job) and “deliver data to my application” (what the programmer needs). Note that TLS sits between transport and application, but we discuss it here because it is intimately tied to TCP and UDP semantics.

Ports: The Transport-Layer Identifier

A port is a 16-bit unsigned integer (0–65 535) that identifies a communication endpoint on a host. Combined with an IP address, it uniquely addresses a process sending or receiving data.

Server side

The server binds to a fixed, well-known port that identifies the service: port 80 = HTTP, port 22 = SSH, port 443 = HTTPS. Clients must know this port in advance.

Client side

The OS assigns a free ephemeral port from a high range (typically 49 152–65 535). This port is temporary, used only for the duration of the connection.

Classification: ports 0–1 023 are well-known (reserved, require elevated privileges); ports 1 024–49 151 are registered; ports 49 152–65 535 are dynamic/ephemeral.

Two Protocols, Two Philosophies

The transport layer offers two fundamental protocols—think of them as two postal services for your data:

TCP — Transmission Control Protocol

Connection-oriented. Before any data flows, TCP establishes a connection via a handshake. It guarantees delivery, preserves ordering, detects duplication and manages flow control. The price: higher overhead and latency.

RFC 793 (1981), updated by RFC 9293 (2022).

UDP — User Datagram Protocol

Connectionless. No handshake, no connection state, no delivery guarantees. Each datagram is independent. The reward: minimal overhead, low latency, simplicity.

RFC 768 (1980)—one of the shortest RFCs ever published.

Neither protocol is universally “better.” The choice depends entirely on the application’s requirements. We examine each in depth, starting with TCP.

TCP: Characteristics

TCP treats data as a continuous byte stream, not as discrete messages. The sending TCP divides this stream into segments, numbers each one, and the receiving TCP reassembles them in order.

What TCP provides

Reliable delivery — every byte is acknowledged; lost segments are retransmitted.

Ordered delivery — out-of-order segments are reassembled using sequence numbers.

Flow control — the receiver advertises available buffer space.

Congestion control — TCP detects network congestion and throttles.

What TCP costs

Setup latency — one round-trip for the three-way handshake.

Header overhead — minimum 20 bytes per segment.

Head-of-line blocking — a lost segment stalls delivery of all subsequent data.

Complexity — TCP is among the most complex kernel subsystems.

The TCP Header (Minimum 20 Bytes)

The sequence number tracks which byte in the stream this segment begins at. The acknowledgement number tells the sender which byte the receiver expects next. Together they enable loss detection, reordering and retransmission.

TCP Flags: Controlling the Connection

The TCP header contains nine single-bit flags, each signalling a specific control action:

Connection lifecycle

SYN — initiate (synchronise sequence numbers).

FIN — graceful close (“no more data”).

RST — force abort (error/rejection).

Data control

ACK — acknowledgement field is valid.

PSH — push to application immediately.

URG — urgent data present (rare).

Congestion signals

ECE — congestion echo.

CWR — congestion window reduced.

NS — nonce concealment (experimental).

The three flags you will encounter most often are SYN, ACK and FIN. We see them in action next.

The Three-Way Handshake

Before any data can flow, TCP must establish a connection. Both sides synchronise their initial sequence numbers (ISNs) so that subsequent segments can be tracked:

Each side proposes a random ISN. The handshake ensures that both client and server agree on their starting sequence numbers before data begins. TCP options (MSS, SACK, window scale) are also negotiated during the SYN segments.

Why Three Steps? Why Not Two?

A common intuition is that two messages should suffice: “I want to connect” and “Connection accepted.” Consider what could go wrong:

Suppose a SYN packet from an earlier, abandoned connection attempt is delayed in the network. It arrives at the server minutes later. With only a two-way handshake, the server would immediately establish a connection that the client never intended. Resources are wasted, and the server waits for data that will never arrive.

Predict: What would happen if TCP used only a two-step handshake (SYN → SYN+ACK)?

Nothing—it would function identically
Old, delayed SYN packets could create ghost connections that waste server resources
Connections would be established twice as fast
Only the server would know the connection exists
Correct. The third step (the client’s ACK) confirms that the client is truly active right now and that the SYN it sent is current, not a ghost from an old session. Without this confirmation, the server would allocate resources for phantom connections. This is the fundamental reason TCP requires three messages: both sides must prove they are alive and synchronised in real time.

Closing a TCP connection is a four-segment exchange, because each direction is closed independently (TCP connections are full-duplex):

An alternative to this graceful four-way closure is a RST (reset), which abruptly tears down the connection. RST is used when an error occurs, when a connection attempt is rejected, or when an application crashes without closing its socket cleanly.

Reliability and Flow Control

Reliability mechanism

Every segment sent must be acknowledged. If the sender does not receive an ACK within a calculated timeout (the Retransmission Timeout, or RTO), it retransmits the segment. Sequence numbers enable the receiver to detect duplicates and reorder segments that arrive out of sequence.

This is the mechanism that ensures your file download receives every byte, even when packets are dropped by congested routers.

Flow control (sliding window)

The receiver advertises a window size in every ACK: the number of bytes it is willing to accept before the sender must pause. This prevents a fast sender from overwhelming a slow receiver.

The window field is 16 bits (max 65 535 bytes). On modern high-speed links, this is far too small—which is why the Window Scaling option exists (we cover it shortly).

Do not confuse flow control with congestion control. Flow control protects the receiver from being overwhelmed. Congestion control protects the network from being overwhelmed. TCP performs both, but they are independent mechanisms.

TCP Checksum and Data Integrity

The 16-bit checksum in the TCP header verifies the integrity of the entire segment: header and data. But it does not compute over the TCP segment alone.

TCP prepends a pseudo-header to the checksum calculation. This pseudo-header includes the source IP, destination IP, protocol number (6 for TCP) and segment length—all drawn from the IP header. The pseudo-header is never transmitted; it exists only for checksum computation.

Including IP addresses in the TCP checksum provides cross-layer integrity: if a router corrupts the destination IP (misrouting the packet), the TCP checksum will fail at the receiver, and the segment will be discarded rather than silently delivered to the wrong host. This is a deliberate violation of strict layer independence—a pragmatic design choice from the early days of TCP/IP (RFC 793).

TCP Socket States

A TCP socket transitions through a set of well-defined states that reflect the connection lifecycle. These states are visible at the OS level (via netstat or ss) and are essential for debugging:

The TIME-WAIT state is frequently misunderstood. After closing, the initiator waits for twice the Maximum Segment Lifetime (2 × MSL, typically 60 seconds) to ensure that any delayed segments from this connection have expired before the same 5-tuple can be reused. This prevents ghost data from a previous connection from polluting a new one.

TCP Options: Extensibility by Design

TCP was designed in 1981 for networks that were orders of magnitude slower than today. Rather than redesigning the protocol, the original authors included an options field that allows negotiation of new features. Options are carried only in the TCP header and are negotiated during the handshake (SYN segments).

Each TCP option has the structure: Kind (1 byte) — Length (1 byte) — Data (variable). Two special single-byte options exist: End of Option List (Kind 0) and NOP (Kind 1, used for alignment/padding).

The most important options for modern networks are MSS, SACK, Window Scaling and Timestamps. We examine each next.

MSS — Maximum Segment Size

MSS declares the largest payload (in bytes) that a TCP endpoint is willing to receive in a single segment. It is sent only in SYN packets during the handshake—it is not negotiated during data transfer.

Relationship to MTU

MSS = MTU − IP header (20 bytes) − TCP header (20 bytes). For standard Ethernet (MTU = 1500), the default MSS is 1460 bytes.

Why it matters

Setting MSS correctly prevents IP fragmentation. If a TCP segment exceeds the path MTU, routers must fragment it—which is inefficient and can cause retransmission problems if any fragment is lost.

If no MSS option is present in the SYN, TCP falls back to the default MSS of 536 bytes (the minimum guaranteed MTU of 576 bytes, minus 40 bytes for IP+TCP headers). This is extremely conservative and wastes bandwidth on modern networks.

SACK — Selective Acknowledgement

Without SACK, TCP acknowledgements are cumulative: an ACK for byte 5000 means “I have received all bytes up to 5000.” If bytes 5001–6000 are lost but 6001–8000 arrive, the sender must retransmit everything from 5001 onwards—even the data that arrived safely.

SACK requires negotiation: the sender includes a SACK Permitted option in its SYN; the receiver then uses SACK blocks in its ACKs to report exactly which byte ranges have been received. This dramatically reduces unnecessary retransmissions on lossy links.

More TCP Options: Window Scaling, Timestamps, Fast Open

Window Scaling

The 16-bit window field caps the advertised window at 65 535 bytes—far too small for modern high-bandwidth, high-latency links. Window scaling (RFC 7323) introduces a shift count (0–14) negotiated in SYN. The effective window becomes: window × 2shift.

With shift=14, the maximum window is ~1 GB.

Timestamps

Two fields: TSval (sender’s timestamp) and TSecr (echo of the peer’s timestamp). Used for accurate RTT estimation and for PAWS (Protection Against Wrapped Sequence numbers) to detect very old segments.

TCP Fast Open

Allows data to be piggybacked on the SYN packet itself, eliminating one round-trip for repeat connections. The server issues a cookie on the first connection; subsequent connections include this cookie in the SYN, permitting data to flow immediately.

Requires support on both ends (RFC 7413).

UDP: The Minimalist Protocol

UDP is the antithesis of TCP. Where TCP provides guarantees, UDP provides speed. Where TCP manages state, UDP is stateless. The fundamental design principle: do as little as possible at the transport layer and let the application decide what it needs.

What UDP provides

Multiplexing via ports — the same port mechanism as TCP.

Optional checksum — data integrity verification (mandatory in IPv6).

Message boundaries — each datagram is a discrete message; the application receives exactly what was sent, with boundaries preserved.

What UDP does NOT provide

No connection setup. No delivery guarantees. No ordering. No flow control. No congestion control. No retransmission.

If a datagram is lost, duplicated or arrives out of order, the application must handle it—or accept the loss.

The UDP Header: 8 Bytes of Simplicity

Length field

Covers the entire datagram: header (8 bytes) + payload. Minimum value is 8 (empty datagram). Maximum theoretical value is 65 535 bytes (limited by the 16-bit field).

Checksum

In IPv4, the UDP checksum is optional (set to 0 if not used). In IPv6, it is mandatory—because IPv6 removed the IP-level checksum, so transport-layer integrity checking is essential.

The choice is always application-driven. A file transfer must be complete—use TCP. A live video stream can tolerate occasional frame drops but cannot tolerate delay—use UDP. A DNS query is tiny and must be fast—use UDP (with TCP as fallback for large responses).

Why TLS Is Needed

TCP and UDP transmit data in clear text. Any device along the network path—a router, a Wi-Fi access point, an ISP’s equipment—can read and even modify the data in transit.

Imagine sending your bank credentials over HTTP (TCP port 80). Every router between your browser and the bank’s server can see your username and password in plain ASCII. In 2014, researchers demonstrated that some public Wi-Fi hotspots actively injected advertisements into unencrypted HTTP responses—modifying content in transit.

TLS (Transport Layer Security) sits between the transport layer and the application layer. It provides three guarantees: confidentiality (encryption), integrity (tamper detection) and authentication (verifying the server’s identity via certificates). Its datagram-oriented counterpart is DTLS (Datagram TLS), used over UDP.

TLS in the Protocol Stack

TLS is strictly speaking not a transport-layer protocol—it operates above TCP (or UDP, in the case of DTLS). However, it is tightly coupled to transport semantics. HTTPS is simply HTTP running through a TLS tunnel over TCP port 443.

TLS consists of several sub-protocols: the Handshake Protocol (key exchange, authentication), the Record Protocol (encryption of application data), the Alert Protocol (error signalling) and the Change Cipher Spec protocol (TLS 1.2 only; removed in TLS 1.3).

TLS 1.3 Handshake (Simplified)

TLS 1.3 (RFC 8446, 2018) dramatically simplified the handshake compared to TLS 1.2, reducing it from two round-trips to one:

The key exchange typically uses Diffie–Hellman over elliptic curves (ECDHE). Both sides generate ephemeral key pairs; the shared secret derived from these keys is used to encrypt all subsequent communication. The server’s certificate proves its identity.

Diffie–Hellman: The Key Exchange Idea

The fundamental challenge of encryption: how do two parties who have never met agree on a secret key, when their only channel of communication is public and observable?

The colour-mixing analogy: Alice and Bob publicly agree on a common colour (say, yellow). Each secretly picks a private colour—Alice picks red, Bob picks blue. They each mix their private colour with the public one and exchange the results. Alice receives Bob’s mix (yellow+blue = green) and adds her red; Bob receives Alice’s mix (yellow+red = orange) and adds his blue. Both arrive at the same final colour—but an eavesdropper who saw only the two intermediate mixes cannot reverse the mixing to discover it.

The mathematical equivalent uses modular exponentiation: the “mixing” operation is easy to compute forward but computationally infeasible to reverse (the discrete logarithm problem). This is the foundation of modern key exchange, first published by Whitfield Diffie and Martin Hellman in 1976.

QUIC: Why a New Transport Protocol?

TCP has served the internet since 1981, but certain limitations have become painful in the modern web:

Head-of-line blocking

TCP guarantees ordered delivery. If a single packet is lost in an HTTP/2 multiplexed connection, all streams on that connection stall—even those whose data arrived intact. This defeats the purpose of multiplexing.

Handshake overhead

A fresh HTTPS connection requires a TCP handshake (1 RTT) plus a TLS handshake (1 RTT in TLS 1.3). That is at least 2 round-trips before any application data flows. On a high-latency link (e.g. 200 ms RTT), this adds 400 ms of pure setup cost.

QUIC (originally “Quick UDP Internet Connections,” now a proper name) was designed by Google and standardised in RFC 9000 (2021) to solve both problems. It runs over UDP, integrates TLS 1.3 encryption natively, and achieves 0-RTT or 1-RTT connection establishment.

QUIC Architecture

Independent streams

QUIC supports multiplexed streams within a single connection. A lost packet on stream A does not block delivery on stream B—eliminating head-of-line blocking.

0-RTT reconnection

For repeat connections, QUIC can send encrypted application data in the very first packet (0-RTT), achieving instant setup for returning visitors.

Knowledge check: What is the primary TCP limitation that motivated the creation of QUIC?

TCP lacks encryption support
Head-of-line blocking across multiplexed streams combined with handshake overhead
TCP cannot carry HTTP traffic efficiently
UDP is inherently faster than TCP in all scenarios
Correct. TCP’s ordered-delivery guarantee means that when a single packet is lost on a multiplexed HTTP/2 connection, all streams are blocked until that packet is retransmitted. Combined with the two-RTT setup cost (TCP + TLS handshakes), this creates measurable latency on the modern web. QUIC solves both: it handles streams independently (no cross-stream blocking) and merges the transport and TLS handshakes into a single round-trip.

Python: A Minimal TCP Server

This is the pattern you will use in Seminar 8 to build an HTTP server from raw sockets. Every line maps to a concept we covered today:

import socket # Configuration HOST = "0.0.0.0" PORT = 8080 BACKLOG = 5 # Create a TCP socket (AF_INET = IPv4, SOCK_STREAM = TCP) server = socket.socket( socket.AF_INET, socket.SOCK_STREAM ) # Allow port reuse (avoids TIME-WAIT issues) server.setsockopt( socket.SOL_SOCKET, socket.SO_REUSEADDR, 1 ) # Bind to address + port, then listen server.bind((HOST, PORT)) server.listen(BACKLOG) print(f"Listening on {HOST}:{PORT}") # Accept one client connection conn, addr = server.accept() print(f"Connected: {addr}") # Receive up to 4096 bytes data = conn.recv(4096) print(data.decode("utf-8")) conn.close() server.close()

Note SO_REUSEADDR: it tells the kernel to allow binding to a port that is in TIME-WAIT state—exactly the state we discussed on slide 21.

Returning to the Streaming Paradox

Remember our opening scenario? Your live football stream froze, but your flatmate’s download was complete. Now you can explain exactly what happened:

The live stream (UDP/RTP)

Live video typically uses UDP (often via RTP/RTSP). When packets are lost, there is no retransmission—the player skips the missing frames. The freeze was likely caused by a burst of packet loss exceeding the player’s buffer. UDP accepted the loss; the application coped as best it could.

The dataset download (TCP)

The download used TCP. When packets were lost, TCP retransmitted them automatically. The application saw a perfect, ordered byte stream—but the retransmissions added latency. TCP’s flow and congestion control further throttled the transfer rate to avoid overwhelming the network.

The transport layer made the choice: the streaming application chose UDP because real-time media prioritises low latency over completeness. The download tool chose TCP because data integrity is non-negotiable for a dataset. The protocol selection is an architectural decision made at design time by the application developer.

Knowledge Check: TCP Mechanics

Question 1: What is the purpose of the three-way handshake?

To transfer data faster
To synchronise initial sequence numbers and confirm bidirectional reachability
To encrypt the connection
To assign port numbers to both endpoints
Correct. The handshake achieves two things: both sides agree on their ISNs (enabling loss detection and reordering) and both prove they are alive and reachable right now (preventing ghost connections from delayed SYN packets).

Question 2: Why does the TIME-WAIT state exist?

To speed up future connections to the same server
To save bandwidth by caching connection parameters
To ensure delayed segments from this connection expire before the 5-tuple is reused
It serves no practical purpose; it is a legacy artefact
Correct. TIME-WAIT lasts 2 × MSL (typically 60 seconds). During this period, any lingering segments from the old connection will expire. Without TIME-WAIT, a new connection using the same 5-tuple could receive stale data from the old one—a silent corruption that would be extremely difficult to diagnose.

Knowledge Check: UDP, TLS and QUIC

Question 3: When is UDP preferable to TCP?

When transferring files that must arrive completely
When low latency matters more than guaranteed delivery (e.g. live streaming, gaming, DNS)
When encryption is required
Always—UDP is the superior protocol
Correct. UDP is the right choice when the application can tolerate some data loss but cannot tolerate the overhead of connection establishment, retransmission delays or head-of-line blocking. DNS queries, real-time audio/video and online games are classic UDP use cases.

Question 4: What security service does TLS provide that neither TCP nor UDP offers?

Reliable delivery
Flow control
Encryption and server authentication
Faster data transfer
Correct. TCP provides reliability and ordering; UDP provides simplicity and low overhead. Neither encrypts data or verifies the identity of the remote peer. TLS adds confidentiality (encryption), integrity (tamper detection via MACs) and authentication (certificates). These are complementary services, not replacements for transport-layer functions.

Key Conclusions

The transport layer’s role

Bridges the network and the application. Provides process identification via ports, and optionally reliability, flow control and congestion control.

TCP: the reliable workhorse

Connection-oriented, byte-stream, reliable, ordered. Three-way handshake, four-way termination, sliding window flow control. Extended by options: MSS, SACK, window scaling, timestamps, Fast Open.

UDP: the minimalist

Connectionless, datagram-based, best-effort. 8-byte header. Application decides what (if anything) to guarantee. Ideal for latency-sensitive and query-response protocols.

TLS and QUIC

TLS adds encryption and authentication above TCP. QUIC combines transport + TLS over UDP, eliminating head-of-line blocking and reducing handshake latency. Foundation of HTTP/3.

Synthesis question: A video-conferencing application uses UDP for audio/video streams but TCP for the text chat. Why does the same application use both transport protocols simultaneously?

It is a programming error—they should both use TCP
