The Casino Aquarium Hack

A documented incident (2017)

A North American casino installed a smart fish-tank thermometer in its lobby. The device monitored water temperature, pH levels and feeding schedules — all transmitted to the cloud via the casino's corporate network.

Attackers compromised the thermometer, used it as a pivot point and exfiltrated 10 GB of high-roller data from the casino's internal database. The breach was detected only when anomalous outbound traffic was flagged by an external threat-intelligence firm.

The question we shall answer today

How did a thermometer provide a pathway into a secured financial database? And what architectural decisions would have prevented it? We shall return to this on slide 34.

When IoT Becomes a Weapon: The Mirai Botnet

One compromised thermometer is unfortunate. What about 600,000 compromised devices?

Mirai (October 2016)

The Mirai malware scanned the internet for IoT devices using factory-default credentials (admin/admin, root/root). It infected cameras, routers and DVRs to build a botnet that launched a 1.2 Tbps DDoS attack against Dyn DNS.

The result: Twitter, Netflix, Reddit, GitHub and Spotify became unreachable across the eastern United States for most of a day.

The missing piece

Every protocol we have studied follows a request–response or client–server pattern. But what happens when 10,000 sensors must push data continuously to consumers who may join or leave at any moment — without knowing who those consumers are?

Checkpoint: Before We Begin

Which transport-layer protocol would you choose for a temperature sensor that sends a 20-byte reading every second?
ATCP with a new connection per message — reliability is always the priority
BTCP with a persistent connection — amortise the handshake over thousands of messages
CUDP — sensor data is best treated as fire-and-forget
DTLS over UDP (DTLS) — encryption is the primary concern
Correct. MQTT, the protocol we study today, runs over a persistent TCP connection. The three-way handshake cost (recall Week 8: SYN → SYN-ACK → ACK) is paid once; subsequent messages travel with minimal overhead. For tiny, frequent payloads, this amortisation is essential.
Not quite. Consider the overhead ratio: a 20-byte payload with a fresh TCP handshake every second wastes bandwidth. Pure UDP avoids the handshake but sacrifices delivery guarantees that matter for actuator commands. The practical solution — and the approach MQTT takes — is a persistent TCP connection established once and reused indefinitely.

The Problem: Why HTTP Polling Fails for IoT

Suppose you manage 500 temperature sensors in a factory. With HTTP, you face two unappealing options:

⏱

Option A: Polling

The server asks each sensor "any new data?" every second. That is 500 HTTP requests/s — each with headers, TCP overhead and wasted roundtrips when nothing has changed.

🔗

Option B: WebSockets

Better, but HTTP was designed for document retrieval, not for routing messages between thousands of producers and consumers. There is no native concept of topics or quality of service levels.

What we need

A protocol that is lightweight (minimal header overhead), push-based (sensors publish when data arrives), and decoupled (publishers and subscribers need not know about each other). This cannot be solved with our existing toolkit.

From Request–Response to Publish–Subscribe

What we know

Client initiates. Server responds. One-to-one. Tightly coupled.

What we need

Publishers push. Subscribers receive. Many-to-many. Fully decoupled.

Part I

IoT Fundamentals

Architecture, the publish/subscribe model and the MQTT protocol

What Is an IoT System?

Definition

An Internet of Things (IoT) system is a distributed system composed of resource-constrained devices that communicate via event-driven messaging, enabling remote observation, control and automation of physical processes.

The defining characteristics go beyond "things with internet access":

📡

Event-driven communication

Data flows when events occur (temperature changes, motion detected) — not on a fixed polling schedule.

🔌

Decoupled components

Producers and consumers operate independently. A sensor need not know who reads its values.

⚙

Automation loops

Sensor data triggers actuator responses (e.g. temperature exceeds 28°C → fan activates) without human intervention.

Generic IoT Architecture

A typical IoT deployment contains four functional components. No component communicates directly with another — all messages flow through the broker.

The Publish/Subscribe Model

IoT communication is asynchronous, message-oriented and spatially and temporally decoupled. What does each form of decoupling mean?

Spatial decoupling

The publisher does not know the network address of any subscriber. The broker handles routing. A new subscriber can appear without the publisher being reconfigured.

Temporal decoupling

Publisher and subscriber need not be online simultaneously. With retained messages (covered on slide 15), a subscriber can receive the last published value even if it connects after the publisher has disconnected.

Synchronisation decoupling

Publishing and receiving are non-blocking operations. A publisher does not wait for subscribers to acknowledge receipt (the broker manages QoS on each link independently).

MQTT: Message Queuing Telemetry Transport

Protocol identity

MQTT is a lightweight publish/subscribe messaging protocol designed for constrained devices and low-bandwidth, high-latency networks. It was created at IBM in 1999 and standardised as OASIS MQTT v3.1.1 (2014) and MQTT v5.0 (2019).

Key technical characteristics:

  • Runs over TCP (port 1883) or TLS (port 8883)
  • Fixed header: only 2 bytes minimum
  • Topic-based routing with wildcards
  • Three Quality of Service (QoS) levels
  • Retained messages and Last Will and Testament (LWT)

Security limitation

MQTT has no built-in encryption. By default, all messages — including credentials — travel in plaintext. TLS must be configured separately on the broker and all clients. We shall formalise this on slide 27.

Why "Message Queuing"?

Despite the name, MQTT is not a traditional message queue. Messages are not stored for retrieval by consumers (except retained messages). The broker routes in real time; it does not persist a backlog.

MQTT Topics and Wildcards

Topics are hierarchical UTF-8 strings separated by /. Two wildcards are available for subscriptions only (never for publishing):

+ — single-level wildcard

Matches exactly one level. factory/+/temperature matches factory/floor1/temperature and factory/floor2/temperature but not factory/floor1/sensor/temperature.

# — multi-level wildcard

Matches zero or more levels. Must be the last character. factory/# matches factory, factory/floor1, factory/floor1/temperature and anything below factory/.

MQTT Quality of Service Levels

QoS governs the delivery guarantee between a single client and the broker — not end-to-end between publisher and subscriber.

QoSNameGuaranteeUse case
0At most onceFire and forget. No acknowledgement. Message may be lost.Ambient telemetry (temperature, humidity) where occasional loss is acceptable
1At least onceBroker acknowledges with PUBACK. Message may be delivered more than once.Sensor alerts, status updates — duplicates are tolerable
2Exactly onceFour-step handshake (PUBLISH → PUBREC → PUBREL → PUBCOMP). No loss, no duplication.Financial transactions, actuator commands where duplication is dangerous

Common misconception

QoS 2 does not guarantee exactly-once delivery across the entire path from publisher to subscriber. It guarantees exactly-once between each client and the broker independently. If the publisher uses QoS 2 but the subscriber uses QoS 0, the subscriber may still miss messages.

Retained Messages and Last Will and Testament

Retained messages

When a message is published with the retain flag set, the broker stores the last message for that topic. Any new subscriber immediately receives the retained message upon subscribing — no need to wait for the next publication.

This solves the "late joiner" problem: a dashboard that connects after the sensor has already published still receives the current state.

Last Will and Testament (LWT)

A client can register a "will" message with the broker at connection time. If the client disconnects unexpectedly (network failure, crash), the broker automatically publishes the LWT message to the specified topic.

This enables presence detection: subscribers know when a device has gone offline without polling.

Answer: The broker stores the last retained message. New subscribers to that topic immediately receive it. The LWT message (if configured) is also published to the LWT topic, informing subscribers that the sensor is offline. Both mechanisms work independently.

Checkpoint: MQTT Misconceptions

A publisher sends a message with QoS 2 to topic sensors/temp. A subscriber is subscribed to sensors/temp with QoS 0. How many times will the subscriber receive the message?
AExactly once — QoS 2 guarantees this end-to-end
BExactly once — the broker upgrades the subscriber's QoS
CAt most once — the effective QoS is the minimum of the two
DZero times — QoS mismatch causes the message to be dropped
Correct. The broker delivers to each subscriber at the minimum of the publisher's QoS and the subscriber's QoS. Since the subscriber requested QoS 0, delivery is "at most once" regardless of what the publisher used. The four-step QoS 2 handshake occurs only between the publisher and the broker.
Not quite. QoS in MQTT is negotiated on each link independently. The broker accepts the message from the publisher with QoS 2 (four-step handshake), but then delivers to each subscriber at the subscriber's requested QoS, which in this case is 0 — "at most once", with no acknowledgement.

Part II

Security and Vulnerability Testing

Why IoT amplifies every risk factor — and what we can do about it

Why Security Is Not Optional

IoT does not introduce new vulnerability types — it amplifies existing ones through four multipliers:

📶

Massive attack surface

Every connected device is a potential entry point. A factory with 500 sensors has 500 attack surfaces instead of one server.

🔒

Weak default configurations

Most IoT devices ship with default credentials and no encryption. Mirai exploited exactly this: 61 default username/password pairs covered the majority of consumer IoT devices in 2016.

🔄

Permanent connectivity

Unlike a laptop that is occasionally offline, IoT devices are always reachable. An unpatched sensor is perpetually exposed.

⚠

Difficult to update

Many IoT devices lack secure over-the-air (OTA) update mechanisms. Firmware patches require physical access or are simply never applied.

What Is a Vulnerability?

Formal definition

A vulnerability is a weakness in an information system, system security procedure, internal control or implementation that could be exploited by a threat source (NIST SP 800-30). It may arise from an implementation error, a misconfiguration or a risky design decision.

Vulnerabilities can lead to four categories of impact:

🚫
Unauthorised access
💻
Arbitrary code execution
📤
Data exfiltration
⏸
Service unavailability

Both. A default password is a design decision by the manufacturer (shipping with a known credential) that becomes a misconfiguration when the operator fails to change it. The Mirai botnet exploited precisely this overlap: the vulnerability existed at the design level, but the misconfiguration at the operational level made it exploitable at scale.

Checkpoint: Scanning Concepts

You run nmap -sV -p 2121 172.20.0.12 against the lab's vsftpd container. Nmap reports the port as "open". What does this technically mean?
AThe TCP three-way handshake completed — a service is listening and accepted the connection
BThe port responded with a SYN-ACK but no service banner was received
CThe port is reachable but a firewall is inspecting the traffic before forwarding
DThe service is running and does not require authentication
Correct. In nmap terminology, "open" means the TCP handshake completed successfully: the target sent SYN-ACK and a connection was established. The -sV flag additionally probes for a version banner (e.g. "220 (vsFTPd 2.3.4)"). "Open" says nothing about authentication requirements — only that a process is listening.
Not quite. Nmap defines three port states: open (handshake completed, service listening), closed (target responded with RST — reachable but nothing listening) and filtered (no response at all, likely blocked by a firewall). "Open" specifically means the full TCP handshake succeeded.
Part III

Formalising IoT Security

Protocol structures, TLS, vulnerability taxonomies and defence-in-depth

MQTT Packet Structure

Every MQTT packet begins with a fixed header of 2–5 bytes. Compare this with HTTP, where headers alone can exceed 500 bytes:

Overhead comparison

MQTT fixed header: 2 bytes. HTTP/1.1 GET request (headers alone): typically 200–800 bytes. For a 20-byte sensor reading, MQTT's overhead is roughly 1:10; HTTP's overhead can exceed 10:1. This is why MQTT dominates constrained-device communication.

Securing MQTT with TLS

Since MQTT has no built-in encryption (recall slide 12), transport security must be provided by TLS. The convention established by IANA assigns port 8883 for MQTT over TLS — sometimes called MQTTS.

Without TLS (port 1883)

A Wireshark capture reveals the full topic name, the message payload and even the username and password in plaintext. This is exactly what OWASP I2 and I7 warn about.

With TLS (port 8883)

Wireshark shows only the TLS handshake and encrypted application data. Topic names, payloads and credentials are invisible to an eavesdropper.

Recall from Week 8

TLS (Transport Layer Security) establishes an encrypted channel via a handshake that negotiates cipher suites, exchanges certificates and derives session keys. The mechanism is identical whether the application protocol is HTTP, SMTP or MQTT — TLS operates at the transport boundary.

Mosquitto broker configuration for TLS:

listener 8883 certfile /certs/server.crt keyfile /certs/server.key cafile /certs/ca.crt require_certificate false

Vulnerability Testing: A Taxonomy

Three progressively deeper activities share similar tools but differ in scope and intent:

🔍

Vulnerability scanning

Automated tool runs against a target to identify known vulnerabilities (CVEs). No exploitation attempted. Example: nmap --script vuln.

📋

Vulnerability assessment

Scanning plus analysis: prioritising vulnerabilities by severity, assessing business impact and recommending remediations. The output is a report, not an exploit.

🎯

Penetration testing

Controlled exploitation of identified vulnerabilities to demonstrate real impact. The tester follows the same workflow as an attacker: reconnaissance → scanning → enumeration → exploitation → reporting.

Legal constraint

Penetration testing without explicit, written authorisation from the system owner is a criminal offence in most jurisdictions, including Romania (Legea 161/2003, art. 42–46). Our seminar lab runs entirely in local Docker containers to ensure legal compliance.

Defence-in-Depth

No single security measure is sufficient. Defence-in-depth applies multiple overlapping controls so that the failure of any one layer does not compromise the entire system.

Hardening means reducing the attack surface: removing unnecessary functionality, applying the principle of least privilege and ensuring that every exposed service is both authenticated and encrypted.

Application: Code and Practice

Python implementations, hardening checklists and the hook resolved

Python: Publishing Sensor Data with MQTT

The course starter kit includes a sensor container that uses paho-mqtt. Here is the core logic:

import paho.mqtt.client as mqtt import time import random # Configuration BROKER = "broker" PORT = 1883 TOPIC = "sensors/temperature" # Create the client and connect client = mqtt.Client( callback_api_version=mqtt.CallbackAPIVersion.VERSION2 ) client.connect(BROKER, PORT, 60) client.loop_start() # Publish readings in an infinite loop while True: value = round( 24.0 + random.uniform(-0.8, 0.8), 2 ) client.publish(TOPIC, str(value)) time.sleep(1)

What to notice

The publisher never addresses a specific consumer. It publishes to a topic; the broker handles distribution. Adding a new subscriber (a dashboard, a logger, an actuator) requires no changes to this code.

Python: A Minimal Port Scanner

In the seminar, you will write this from scratch. It combines TCP socket knowledge from Week 2 with the security concepts from today:

import socket TARGET = "172.20.0.10" TIMEOUT = 0.2 def scan_port(ip: str, port: int) -> bool: """Attempt a TCP connect to (ip, port).""" s = socket.socket( socket.AF_INET, socket.SOCK_STREAM ) s.settimeout(TIMEOUT) # connect_ex returns 0 on success result = s.connect_ex((ip, port)) s.close() return result == 0 # Scan ports 1 to 1024 for port in range(1, 1025): if scan_port(TARGET, port): print(f"[OPEN] {port}")

Limitations of this approach

This scanner detects only TCP ports in the "open" state. It cannot detect UDP services, stealth-scan filtered ports or perform version fingerprinting. For those capabilities, you will use nmap in the seminar.

The Casino Revisited: What Went Wrong

We now have the vocabulary to diagnose the aquarium hack from slide 2:

🐟

Flat network (no segmentation)

The thermometer sat on the same network segment as the high-roller database. Once the device was compromised, lateral movement required no additional barrier. Defence-in-depth Layer 1 (slide 30) was absent.

🔓

Default credentials (OWASP I1)

The thermometer likely shipped with factory-default credentials and was never reconfigured. The exact vector Mirai exploited at scale, here exploited against a single target.

📤

No egress monitoring

10 GB of data were exfiltrated before detection. The casino had no outbound traffic analysis on its IoT segment — because there was no separate IoT segment to monitor.

The fix

Network segmentation (isolate IoT devices on their own VLAN), mandatory credential rotation, TLS on all IoT communication and egress monitoring on the IoT segment. Every countermeasure maps to one of the defence-in-depth layers.

Hardening Measures: A Practical Checklist

Hardening means reducing the attack surface systematically. For each measure below, predict which defence-in-depth layer it belongs to before revealing the answer.

Change all default credentials

Authentication & ACLs — the innermost defensive ring. Eliminates OWASP I1.

Enable TLS on MQTT (port 8883)

TLS encryption — the second ring. Prevents eavesdropping (OWASP I2, I7). Recall our Mosquitto configuration from slide 27.

Place IoT devices on a separate VLAN

Network segmentation — the outermost ring. Limits lateral movement even if a device is compromised. This single measure would have stopped the casino aquarium hack.

Validate all MQTT payloads

Input validation — the innermost core. Prevents injection attacks via crafted MQTT messages targeting subscribers that parse payload data.

Applying Security Concepts

A factory's MQTT broker is configured on port 1883 (plaintext). An attacker on the same network segment runs Wireshark. Which of the following can the attacker observe?
AOnly the IP addresses and port numbers — MQTT encrypts the application payload by default
BTopic names only — message payloads are encrypted at the broker level
CEverything: topic names, message payloads, client IDs and credentials (if sent)
DNothing useful — MQTT uses application-layer obfuscation even without TLS
Correct. On port 1883 (no TLS), MQTT is entirely plaintext. A Wireshark capture reveals topic names, payload data, client identifiers, CONNECT packets with usernames and passwords — everything. This is precisely OWASP I2 (insecure network services) and I7 (insecure data transfer). The mitigation is TLS on port 8883.
Not quite. MQTT has no built-in encryption or obfuscation. Without TLS, every byte is visible in a network capture. This includes credentials in the CONNECT packet, topic names in PUBLISH packets and the full message payload. You will verify this yourself using Wireshark in the seminar.

Key Takeaways

📡

IoT = distributed + decoupled

IoT systems use publish/subscribe communication through a broker. Components are spatially, temporally and synchronously decoupled.

📨

MQTT is minimal by design

2-byte minimum header, topic-based routing, three QoS levels. Runs over TCP (1883) or TLS (8883). No built-in security.

⚠

IoT amplifies risk

Massive attack surface, default credentials, permanent connectivity and difficult patching multiply every vulnerability.

🛡

Defence-in-depth is the answer

Network segmentation, TLS encryption, authentication/ACLs and input validation — multiple overlapping layers, not a single magic solution.

Synthesis: Bringing It Together

An IoT deployment has 200 sensors on a flat network with default credentials, communicating via MQTT on port 1883. Which single change would produce the greatest security improvement?
ASwitch from QoS 0 to QoS 2 for all messages
BAdd retained messages so the dashboard always has the latest state
CPlace all IoT devices on a separate network segment with firewall rules
DUpgrade to MQTT v5.0 for enhanced authentication
Correct. Network segmentation addresses the broadest category of risk: it prevents lateral movement (the casino hack), limits the blast radius of any compromise and creates a chokepoint where monitoring can be applied. QoS changes affect reliability, not security. MQTT v5.0 improves authentication but does not prevent network-level attacks. Segmentation is the most impactful single intervention.
Consider which change addresses the most attack vectors simultaneously. QoS governs delivery guarantees, not security. Retained messages are a feature, not a defence. MQTT v5.0 authentication helps but only secures the broker connection. Network segmentation limits lateral movement, creates monitoring boundaries and reduces the attack surface across all 200 devices at once — the most impactful single change.
