24 February 2008: YouTube Vanishes

A single misconfiguration, two-thirds of the planet affected

The Pakistani government orders Pakistan Telecom to block YouTube nationally. A junior engineer responds by advertising a false route — claiming Pakistan Telecom owns YouTube's IP address space. The false advertisement propagates through the global routing system. Within minutes, YouTube becomes unreachable for roughly two-thirds of the internet, for nearly two hours.

🔌
Was the cable cut?

No. Physical infrastructure was intact everywhere.

🌐
Was YouTube's server down?

No. The servers were running normally.

🗺️
So what broke?

A single configuration error at one specific layer of the network stack.

The question for today: Which layer? How would you even begin to diagnose it? By the end of this lecture, you will be able to identify the exact layer, explain why the damage propagated so widely and describe the mechanism that should have prevented it.

Why This Matters for You

From outages to everyday debugging

The Pakistan/YouTube incident is dramatic, but the same principle applies to everyday programming. Consider a scenario you will face this semester:

Your Python application connects to a database container. Suddenly, it stops responding. Where is the fault? A disconnected cable? A misconfigured IP? A firewall rule? A crashed process? A socket timeout?

Without a structured mental model, debugging becomes guesswork. Architectural models give you that structure. They decompose the overwhelming complexity of networked systems into manageable layers, each with defined responsibilities.

What you gain today

Recall from Week 1

The concepts we already have in place

Last week, we established four concepts that serve as the entry point for today's lecture. We used them informally; today we formalise them within a layered model.

Protocol

A set of rules governing communication between two entities. We saw that HTTP, TCP and Ethernet are all protocols — but at different levels of abstraction.

Protocol stack

Protocols organised in layers, each building on the services of the one below. We noted that your browser relies on a stack of protocols, not just one.

Transfer unit

The chunk of data handled at a given layer — we mentioned packets and frames without yet defining the distinction precisely.

Encapsulation

The process of wrapping data with headers as it moves down the stack. We observed this in Wireshark during the first seminar.

Today's goal: transform these informal intuitions into a precise, layered framework — first through the OSI model (7 layers), then through the TCP/IP model (4 layers).

The Complexity Problem

Why networking cannot work without structured decomposition

The challenge: diverse hardware from hundreds of vendors, varied operating systems, dozens of transmission media and thousands of application protocols — all of which must interoperate globally, reliably, at scale.

Without layering

  • Every application must understand every network technology
  • Changing the physical medium means rewriting application software
  • No interoperability between vendors
  • Testing requires examining all combinations — a combinatorial explosion

Result: isolated, proprietary islands. This was the reality of the 1970s.

With layering

  • Each layer has focused, well-defined responsibilities
  • Layers interact through stable interfaces
  • Components can be replaced without affecting other layers
  • Vendors can innovate independently at each layer

Result: a global Internet connecting trillions of devices from thousands of manufacturers.

Thought Experiment: A World Without Layers

What happens when we remove the architecture?

Suppose you write a chat application. Without layering, your code must handle everything:

Your responsibilities (no layers)

  • Encode each character into electrical voltages
  • Handle collisions on the wire
  • Determine the physical route through every router
  • Detect and correct transmission errors bit by bit
  • Manage connection state, retransmissions and ordering
  • Encrypt the data and negotiate ciphers
  • Render the user interface

Your responsibilities (with layers)

  • Create a socket
  • Connect to the server
  • Send and receive messages
  • Close the connection

The operating system and network hardware handle everything else — because the responsibilities are separated into layers.

A web developer writes HTTP. The OS handles TCP. The network card handles Ethernet. The cable carries signals. None of them needs to understand the others' internals. This is the power of layered architecture — and it is exactly what the OSI and TCP/IP models describe.

The Construction Analogy

Understanding layered architecture through a familiar structure

Separation of concerns: an electrician does not alter the foundations. An interior designer does not rewire the building. Each specialist works at their own layer.

  • Upper layers rely on services provided by lower layers
  • Changes in one layer do not require rebuilding the entire structure
  • Each layer can be designed by different specialists (or vendors)

In networking: a web developer writes HTTP without understanding optical fibre. A hardware engineer designs network cards without understanding HTTP. The layered model makes this possible.

The OSI Reference Model

Open Systems Interconnection — the theoretical framework

OSI = Open Systems Interconnection, standardised by ISO as ISO 7498 (1984).

Characteristics

  • A theoretical reference model — not an implementation
  • Seven distinct layers, each with a precisely defined role
  • Developed as a response to the chaos of proprietary systems (IPX/SPX, DECnet, SNA)
  • Specification is independent of any hardware or operating system

Its role today

OSI was never widely implemented as a complete protocol suite — TCP/IP won that race. Yet OSI remains the standard common vocabulary for discussing networks. When an engineer says "that is a Layer 3 problem", they are referencing the OSI model.

OSI is a reference model. TCP/IP is an implementation model. We need both: one for thinking, one for building.

Layers 1 & 2 — Physical and Data Link

The foundation of all network communication

Layer 1 — Physical

  • Role: transmit raw bits over a physical medium
  • PDU: bit (or symbol)
  • Concerns: voltage levels, signal modulation, bit-level synchronisation, connectors
  • Examples: Ethernet cables (Cat5e/6), optical fibres, Wi-Fi radio

Key standard: IEEE 802.3 defines Ethernet at the physical layer.

Layer 2 — Data Link

  • Role: reliable node-to-node frame delivery and error detection
  • PDU: frame
  • Addressing: MAC addresses (48 bits, e.g. 00:1A:2B:3C:4D:5E)
  • Sublayers: LLC (Logical Link Control) and MAC (Media Access Control)
  • Functions: framing, physical addressing, error detection (CRC), flow control

Scope: Layer 2 operates within a single network segment (LAN).

Layer 3 — Network

Logical addressing and routing across network boundaries

AspectDetail
RoleRoute packets between different networks
PDUPacket
AddressingIP addresses (32 bits for IPv4, 128 bits for IPv6)
Key protocolsIP, ICMP, ARP, routing protocols (OSPF, BGP)

Primary responsibilities

  • Logical addressing: IP addresses identify hosts globally
  • Routing: determining paths across networks
  • Fragmentation: splitting packets for smaller MTU links

MAC vs IP: a critical distinction

MAC (Layer 2)IP (Layer 3)
Physical (hardware-assigned)Logical (configurable)
Local scope (same LAN)Global scope (routable)
Changes at each hopRemains constant end-to-end

At each router hop, the source and destination MAC addresses change (they point to the current hop and the next hop), but the IP addresses remain constant from sender to receiver. This distinction is one of the most commonly misunderstood concepts.

Knowledge Check

A common point of confusion about addressing

Scenario: Your laptop sends an HTTP request to a server three router hops away. Which statement about addressing is correct?

Correct. IP addresses (Layer 3) identify the ultimate source and destination and remain constant throughout the journey. MAC addresses (Layer 2) identify the current and next hop on the local link and are rewritten by each router. This is why we need both addressing schemes: IP for global routing, MAC for local delivery.

Not quite. This is one of the most common misconceptions in networking. The IP addresses stay fixed (they name the endpoints), while the MAC addresses change at every router (they name the "next door neighbour" at each step). Think of it as a postal system: the destination address on the letter (IP) never changes, but the delivery van (MAC) changes at each distribution centre.

Layer 4 — Transport

End-to-end communication between processes

TCP — Transmission Control Protocol

  • Connection-oriented: three-way handshake (SYN, SYN-ACK, ACK)
  • Reliable: acknowledgements and retransmissions
  • Ordered: data delivered in the sequence sent
  • Flow and congestion control: built in

Use when every byte must arrive correctly: HTTP, SSH, email

UDP — User Datagram Protocol

  • Connectionless: no handshake, no persistent state
  • Unreliable: no delivery guarantees
  • Unordered: packets may arrive out of sequence
  • Minimal overhead: 8-byte header (vs 20+ bytes for TCP)

Use when speed and low latency matter: DNS, VoIP, gaming

Socket = IP address + port number. This combination uniquely identifies an application endpoint. Port numbers range from 0 to 65 535 (16 bits). We will programme sockets directly starting in Week 3.

Layers 5, 6 and 7 — The Upper Layers

Session, Presentation and Application

Layer 5 — Session

Manages the dialogue between applications: establishment, maintenance and termination of sessions. Controls half-duplex or full-duplex communication.

Examples: NetBIOS, RPC

Layer 6 — Presentation

Translates data formats, handles encryption/decryption and compression. Ensures both sides interpret the data identically.

Examples: TLS/SSL (conceptually), JPEG, UTF-8, ASN.1

Layer 7 — Application

The interface between the network and the user or application. Provides protocols for file transfer, email, web browsing and name resolution.

Examples: HTTP, FTP, SMTP, DNS, SSH

In practice, the boundaries between layers 5, 6 and 7 are blurred. Most modern applications handle all three concerns internally. This is precisely why the TCP/IP model collapses them into a single "Application" layer — a pragmatic decision we shall examine shortly.

How Layers Communicate

Vertical data flow, virtual horizontal communication

Fundamental principles

  • A layer offers services to the layer directly above
  • A layer uses services from the layer directly below
  • Each layer "sees" only adjacent layers — not the full stack
  • Communication between peer layers is virtual: data physically flow down, across the medium and back up

Service Access Point (SAP): the interface through which one layer makes its services available to the layer above. In TCP/IP, the socket API is the SAP between the application and the transport layer.

Key terminology

TermMeaning
PDUProtocol Data Unit — the complete unit of data at a given layer
SDUService Data Unit — the payload received from the layer above
HeaderControl information prepended by each layer
TrailerControl information appended (typically by Layer 2 only, for CRC)

The relationship is: PDU = Header + SDU (+ Trailer where applicable).

Watch data gain headers as they descend through the stack

Suppose your browser sends an HTTP GET request. At the application layer, this is raw data. As it passes down through each layer, it is wrapped with additional control information:

Each layer's header is opaque to the layers below. The IP layer does not inspect the TCP header; Ethernet does not inspect the IP header. This opacity is what allows independent development and replacement of protocols at each layer.

Decapsulation — The Reverse Process

At the receiver, headers are stripped layer by layer

When the frame arrives at the destination host, each layer removes its own header (a process called decapsulation) and passes the remaining payload to the layer above:

Sender (encapsulation)

  • L7: generates HTTP data
  • L4: adds TCP header → segment
  • L3: adds IP header → packet
  • L2: adds Ethernet header + FCS trailer → frame
  • L1: converts to electrical/optical signals

Receiver (decapsulation)

  • L1: receives signals, reconstructs bits
  • L2: reads Ethernet header, verifies FCS, strips both
  • L3: reads IP header, strips it
  • L4: reads TCP header, strips it
  • L7: receives the original HTTP data

A router operates at Layer 3. It decapsulates only up to the IP header (to read the destination address and make a routing decision), then re-encapsulates with a new Layer 2 frame for the next hop. The IP header and everything above it pass through unchanged. This is why MAC addresses change at each hop but IP addresses remain constant.

Where Layers Live

Hardware, operating system and application responsibilities

The socket API sits at the boundary between the application (your code) and the OS kernel (transport + network layers). When you call socket.send() in Python, you hand data to the kernel, which handles TCP segmentation, IP routing and passes it down to the NIC driver. This is exactly what you will practise in this week's seminar.

TCP/IP = Transmission Control Protocol / Internet Protocol. Developed by DARPA in the 1970s, standardised through RFCs (Request for Comments).

Key differences from OSI

  • Four layers instead of seven
  • Designed around existing protocols, not as an abstract specification
  • Session and Presentation are not separate layers
  • The Internet layer is connectionless by design

The Protocol Wars

How pragmatism defeated bureaucracy

In the 1980s, the telecommunications industry backed the OSI model — designed by ISO committees and documented in volumes measured by weight. Meanwhile, the academic community promoted TCP/IP with a different philosophy: "rough consensus and running code" (Dave Clark, IETF, 1992).

OSI took seven years from specification to first implementation. TCP/IP shipped with BSD UNIX in 1983 — free, tested at scale on ARPANET, and immediately useful. Universities planned to adopt OSI once it became ready, then quietly abandoned the plan.

OSI's burden

  • Seven years from design to first implementation
  • Specification documents measured in kilogrammes
  • Designed by committees, not by working engineers

TCP/IP's advantage

  • Free reference implementation in BSD UNIX 4.2 (1983)
  • Tested at scale before formal standardisation
  • "The code is the specification"

Why We Teach Both Models

They serve complementary purposes

Use OSI when you need to…

  • Analyse a network problem layer by layer
  • Discuss responsibilities with precision ("that is a Layer 2 issue")
  • Learn the full scope of network functions
  • Troubleshoot systematically from the bottom up

Use TCP/IP when you need to…

  • Implement a protocol or network service
  • Programme applications using the socket API
  • Configure real systems and infrastructure
  • Read RFCs and technical documentation

In professional practice, engineers freely mix terminology from both models. You might hear "that is an L3 problem" (OSI vocabulary) in a conversation about configuring IP routing (TCP/IP context). This is normal and expected — the models are complementary, not competing.

Knowledge Check

Testing your understanding of encapsulation

Question: When your browser sends an HTTP request, in what order are protocol headers added during encapsulation?

Correct. Data originates at the Application layer (HTTP), then flows down the stack. The Transport layer adds the TCP header, the Network layer adds the IP header and the Data Link layer adds the Ethernet header and FCS trailer. The order follows the stack top-to-bottom: HTTP → TCP → IP → Ethernet.

Not quite. A common mistake is thinking headers are added bottom-up (from physical to application). In reality, the application generates data first, and each lower layer wraps it with its own header. The correct order is: the application produces HTTP data → TCP adds its header → IP adds its header → Ethernet adds header and trailer. Think of encapsulation as putting a letter (HTTP) into progressively larger envelopes.

Formal Definitions

Precise terminology for the concepts we have been building

Protocol Data Unit (PDU): the unit of data specified by a protocol at a given layer. Each layer's PDU consists of the header added by that layer plus the SDU it received from above.

Service Data Unit (SDU): the payload delivered from the layer above. From the perspective of Layer N, the SDU is the PDU of Layer N+1.

Service Access Point (SAP): the logical interface through which a layer offers services to the layer above. In TCP/IP, port numbers serve as SAPs at the transport layer.

The PDU relationship

PDUN = HeaderN + SDUN

where SDUN = PDUN+1

Example: at Layer 3, the PDU is a packet. The packet consists of the IP header plus the SDU, which is the TCP segment (i.e. the PDU of Layer 4).

Limitations of the OSI Model

Where theory meets the messy reality of implementation

The OSI model is a powerful analytical tool, but it is not a perfect description of reality. Understanding its limitations prevents common mistakes:

What OSI gets wrong (or oversimplifies)

  • Layers 5 and 6 are rarely separate in practice — most protocols handle session and presentation concerns within the application itself
  • Some protocols span multiple OSI layers (e.g. TLS operates between L4 and L7)
  • The model was designed before the protocols were tested at scale, leading to some layer definitions that do not map neatly to real implementations

What OSI gets right

  • The principle of layered abstraction is universally applicable
  • The vocabulary (Layer 2, Layer 3 etc.) is used daily by every network engineer
  • The encapsulation model accurately describes how data are processed
  • The debugging methodology (bottom-up, layer-by-layer) is indispensable

In practice, most engineers use a hybrid mental model: the four TCP/IP layers for implementation and configuration, with OSI layer numbers for precise discussion. You will hear "L2 switch" (an OSI term) and "IP routing" (a TCP/IP term) in the same sentence. The goal is clarity, not doctrinal purity.

From Theory to Code

Introduction to the Socket API

Connecting question: We now understand the layers — but how do we, as programmers, actually use them? The answer is the socket API.

The Socket API

Sockets provide the programming interface between your application (Layer 7) and the transport layer (Layer 4). The operating system handles everything below.

Socket: an endpoint for network communication, identified by an IP address and port number. The API allows creating connections and exchanging data over TCP or UDP.

A brief history

The socket API was introduced in BSD UNIX 4.2 (1983) by researchers at the University of California, Berkeley. It became the de facto standard for network programming across virtually every operating system: Linux, Windows (Winsock) and macOS.

The API you will use in Python's socket module is a thin wrapper around these same BSD socket system calls.

Socket Types

Choosing the correct socket for your needs

SOCK_STREAM (TCP)

A Minimal TCP Server in Python

The server-side socket sequence: socket → bind → listen → accept

import socket

# Create a TCP socket (IPv4)
sock = socket.socket(
    socket.AF_INET,
    socket.SOCK_STREAM
)

# Allow immediate address reuse after restart
sock.setsockopt(
    socket.SOL_SOCKET,
    socket.SO_REUSEADDR,
    1
)

# Bind to all interfaces, port 9090
sock.bind(("0.0.0.0", 9090))

# Start listening (backlog of 32)
sock.listen(32)

while True:
    # Block until a client connects
    conn, addr = sock.accept()
    data = conn.recv(1024)
    response = b"OK: " + data.upper()
    conn.sendall(response)
    conn.close()

TCP Three-Way Handshake

Establishing a reliable connection

Why three messages? Each side must confirm that it can both send and receive. SYN initiates client→server, SYN-ACK confirms server→client and the final ACK confirms client→server receipt. Only after this three-step process do both sides have confidence that the connection is bidirectional. We shall explore TCP in detail in Week 8.

Layer-by-Layer Debugging

A systematic approach using the OSI model

When a network application fails, work from the bottom up through the layers. Confirm each layer is functional before investigating the next:

LayerDiagnostic questionCommand
L1 PhysicalCable connected? Link LED on?ip link show
L2 Data LinkInterface up? Correct MAC?ip link show, arp -a
L3 NetworkIP address assigned? Gateway reachable?ip addr, ping, traceroute
L4 TransportPort open? Service listening?ss -tuln, nc -zv host port
L7 ApplicationApplication responds correctly?curl, application logs

This methodology is not optional — it is how professionals debug. Skipping layers leads to wasted time. If L1 is down (cable unplugged), no amount of application-level debugging will help. Always start at the bottom.

Knowledge Check

Apply the layer model to diagnose a real problem

Scenario: Your Python server is running, ss -tuln confirms port 9090 is open, but a colleague on another subnet cannot connect. ping from your machine to theirs succeeds. What is the most likely layer at which the problem lies?

Correct. Ping succeeds, so Layers 1–3 are functional (physical connectivity and IP routing work). The server is listening (confirmed by ss), so the application is running. The most likely culprit is a firewall rule blocking TCP traffic on port 9090 — a Layer 4 (transport) issue. The diagnostic command would be iptables -L or checking the security group / firewall configuration.

Not quite. Let us apply the bottom-up methodology: L1/L2 are working (ping implies physical connectivity). L3 is working (ping succeeds, meaning IP routing is fine). The server is running and listening. The gap is between "IP connectivity works" (L3) and "application cannot be reached" (L7) — that gap is Layer 4. A firewall blocking port 9090 is the most likely cause. This is a classic transport-layer issue.

Resolving the Hook: The Pakistan/YouTube Incident

Now we have the vocabulary to understand what happened

At the start of this lecture, we asked: which layer broke, and why did the damage spread globally? With the layered model in hand, we can now answer precisely:

The fault: Layer 3 — Network

Pakistan Telecom announced a false BGP route, claiming to own YouTube's IP address prefix (208.65.153.0/24). BGP (Border Gateway Protocol) is a Layer 3 routing protocol. The error was not physical (L1), not a framing issue (L2), not a transport failure (L4) and not a bug in YouTube's application (L7).

Why it propagated

BGP routes propagate globally by design — routers share routing information with their peers. Without proper route filtering (a mechanism that should have caught the error), the false announcement spread within minutes. Every router in the affected path began sending YouTube-bound traffic to Pakistan instead of to YouTube's actual servers.

The broader lesson: understanding which layer a problem belongs to is not merely academic — it determines what tools you use, what expertise is needed and how quickly you can diagnose the issue. Layer-based thinking is the single most transferable skill in networking.

Synthesis Challenge

Bringing together everything from today's lecture

Question: A colleague tells you: "I changed the Ethernet cable but my HTTP requests still time out — the problem must be in my Python code." Using the layered model, what is wrong with this reasoning?

Correct. The colleague jumped from Layer 1 (cable) directly to Layer 7 (Python code), skipping Layers 2, 3 and 4 entirely. The problem could be a DHCP failure (no IP address — L3), a firewall rule (L4), a DNS resolution failure (L7 but a different protocol), or any number of intermediate issues. The systematic approach is to verify each layer in sequence before concluding the fault is in application code.

Not quite. The core error in the colleague's reasoning is skipping layers. Replacing the cable addresses Layer 1, and the Python code is at Layer 7. But between these sit Layers 2, 3 and 4 — any of which could be the actual cause (a misconfigured IP address, a firewall blocking the port, a routing issue). The layered model teaches us to verify each layer sequentially, never to jump from bottom to top.

Common Misconceptions — Addressed

Four errors to eliminate from your mental model

❌ "Data travel horizontally between peer layers."

✓ Data flow down the stack at the sender, across the physical medium and then up the stack at the receiver. Horizontal (peer-to-peer) communication is virtual — it exists logically but not physically.

❌ "TCP guarantees delivery."

✓ TCP guarantees delivery or notification of failure. If the network is persistently unreachable, TCP will eventually time out and report an error. Guaranteed delivery would require solving the Two Generals Problem, which is provably impossible.

❌ "The OSI model is used in real networks."

✓ TCP/IP is the implementation model used in practice. OSI is a reference model — its value lies in providing a common analytical vocabulary, not in being a blueprint for software.

❌ "MAC addresses do not change during routing."

✓ Source and destination MAC addresses change at every router hop. They identify the current and next node on the local link. IP addresses (normally) remain constant end-to-end.
